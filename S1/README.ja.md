**ラボA — Wireshark：はじめてのキャプチャ**

業界標準のパケットスニファ **Wireshark** を実際に触ります。ツールをインストールし、ネットワークインターフェイス上でライブキャプチャを開始し、平文の `http://` ページを読み込んでトラフィックを発生させ、表示フィルタで **HTTP `GET`** リクエストを絞り込み、そのOSI各層（Ethernet → IP → TCP → HTTP）をたどってカプセル化を実際に確認します。最後にキャプチャを `.pcap` ファイルとして保存します。さらにボーナスの **ARP アクティビティ** で、コンピュータがIPをMACアドレスにどう解決するかを見ます。

**ラボB — Cisco Packet Tracer：はじめてのネットワーク構築**

無料のネットワークシミュレータ **Cisco Packet Tracer** でネットワークをゼロから構築します。2台のPCと **2901 ルータ** を配置し、ケーブルでつなぎ、2つの*異なる*サブネットを設定して、ネットワーク間でトラフィックを動かすにはなぜ（スイッチではなく）ルータが必要かを証明します。IPを割り当ててCLIでルータインターフェイスを起動した後、`ipconfig` とサブネットをまたぐ `ping` で確認し、**シミュレーションモード**で再生してカプセル化がホップごとに起きる様子を観察します。さらにボーナスの **ARP アクティビティ** で、ホストがサブネットをまたいで送信する前にゲートウェイのMACをどう解決するかを明らかにします。

### 詳解：`ping` の前に `ipconfig` でネットワーク構成を読む

何かを意味あるかたちで `ping` する前に、**自分のアドレス・ゲートウェイ・サブネット**を知る必要があります。`ipconfig` 系のコマンドはまさにそれを表示します — マシンがネットワークのどこに位置するかを把握するために、どのアナリストも*最初に*打つコマンドです。

| OS | コマンド | 補足 |
| :--- | :--- | :--- |
| **Windows** | `ipconfig` | MACアドレス・DNSサーバ・DHCPリース情報には `/all` を付加。 |
| **macOS** | `ifconfig` | Wi-FiのIPだけなら `ipconfig getifaddr en0`、概要は `networksetup -getinfo Wi-Fi`。 |
| **Linux** | `ip addr`（`ip a`） | 古い環境では `ifconfig`。デフォルトゲートウェイは `ip route` で確認。 |

#### `ipconfig` 出力フィールドの読み方
```text
   IPv4 Address. . . . . . . . . . . : 192.168.1.37
   Subnet Mask . . . . . . . . . . . : 255.255.255.0
   Default Gateway . . . . . . . . . : 192.168.1.1
   DNS Servers . . . . . . . . . . . : 192.168.1.1
```
* **`IPv4 Address`（`192.168.1.37`）**: ローカルネットワーク上の*自分の*ホストアドレス — 送信するすべての送信元IP。
* **`Subnet Mask`（`255.255.255.0`）**: アドレスを**ネットワーク部**と**ホスト部**に分けます。ここでは `/24` なので最初の3オクテット（`192.168.1`）がネットワーク — つまり `192.168.1.x` のホストはすべて**ローカル**です。
* **`Default Gateway`（`192.168.1.1`）**: **ルータ**です。宛先が自分のローカルネットワークに*ない*パケットはすべてここに渡されて先へ転送されます。
* **`DNS Servers`（`192.168.1.1`）**: 名前（`google.com`）をIPアドレスに変換する役割。`ping <名前>` は失敗するのに `ping <IP>` は成功するなら、DNSを疑いましょう。

#### `ping` の前に分析する：宛先はローカルかリモートか

サブネットマスクを当てはめてパケットが*どう*進むかを判断します — これは `ping` が実際に何を試験するかを予測します:

| あなたが `192.168.1.37 /24`、ping先… | 同一ネットワーク？ | パケットが通る経路 |
| :--- | :--- | :--- |
| `192.168.1.50` | ✅ はい（`192.168.1.x`） | **直接** — ARPでMACを解決し、ローカルリンク上でフレーム送信（ルータ不要）。 |
| `10.0.0.5` | ❌ いいえ | **ゲートウェイ経由** — `192.168.1.1` のMAC宛に送られ、そこから転送。 |
| `8.8.8.8` | ❌ いいえ | **ゲートウェイ → インターネット経由** — ルーティング・NAT・ISPを試験。 |

> [!TIP]
> これが [ボーナスARPアクティビティ](./WIRESHARK_GUIDE.ja.md#アクティビティ3--arpこのipのmacは誰) が重要な*理由*です。**ローカル**の宛先ならPCは宛先自身をARPし、**リモート**の宛先なら**ゲートウェイ**をARPします。先に `ipconfig` を読めばどちらのケースか分かります。

#### トラブルシューティングのはしご：何を先に `ping` するか

接続が壊れたら、**1段ずつ**外側へ `ping` します — 各段が異なる層を切り分けるので、最初に失敗した段が問題箇所を指し示します:

1. **`ping 127.0.0.1`**（ループバック）→ このマシンの **TCP/IP スタック**はそもそも動いているか？
2. **`ping <自分のIPv4>`**（例 `192.168.1.37`）→ 自分の**ネットワークアダプタ**は正しく設定されているか？
3. **`ping <デフォルトゲートウェイ>`**（例 `192.168.1.1`）→ **ローカルルータ**に到達できるか？（L1/L2/ローカルサブネットの確認）
4. **`ping 8.8.8.8`**（パブリックIP）→ **インターネットへのルーティング＋NAT**は動いているか？
5. **`ping google.com`**（名前）→ **DNS**は名前を解決しているか？（4は成功して5が失敗なら、接続ではなくDNSの問題）

> [!NOTE]
> はしごを登れば「インターネットがダウンしている」を正確な診断に変えられます。3で失敗 → ローカルネットワークの問題、4までは成功で5が失敗 → DNSの問題。

### 詳解：`ping` コマンドとオプション

`ping` はホストの到達性を確認する最も広く使われるネットワークユーティリティです。OSIモデルの **第3層（ネットワーク層）** で動作する **ICMP（Internet Control Message Protocol）** を利用します。

`ping` を実行すると、システムは宛先IPに **ICMP Echo Request（Type 8）** パケットを送ります。宛先が生きていて到達可能なら、**ICMP Echo Reply（Type 0）** で応答します。

#### `ping` 出力フィールドの分析
```text
64 bytes from 8.8.8.8: icmp_seq=1 ttl=115 time=14.2 ms
```
* **`64 bytes`**: 送信したICMPペイロードのサイズ。
* **`icmp_seq=1`**: パケットのシーケンス番号。パケットロスや順序を追跡できます。
* **`ttl=115`（Time to Live）**: ルータのホップごとに1ずつ減るカウンタ。ルーティングループを防ぐためのもの。0になるとパケットは破棄されます。
* **`time=14.2 ms`（ラウンドトリップタイム / RTT）**: リクエストが宛先に届き応答が戻るまでにかかった総ミリ秒。

#### 重要な `ping` コマンドオプション

| 操作 / 目的 | macOS / Linux コマンド | Windows コマンド | 説明 |
| :--- | :--- | :--- | :--- |
| **パケット数を指定** | `ping -c 5 8.8.8.8` | `ping -n 5 8.8.8.8` | ちょうど `5` 回送信した後に自動で停止。 |
| **連続ping** | `ping 8.8.8.8` *(既定)* | `ping -t 8.8.8.8` | **`Ctrl + C`** で手動停止するまで延々と送信。 |
| **パケットサイズ変更** | `ping -s 1000 8.8.8.8` | `ping -l 1000 8.8.8.8` | カスタムペイロードサイズ（1000バイト）を送り、MTU限界を試験。 |
| **間隔を調整** | `ping -i 0.5 8.8.8.8` | *ネイティブ非対応* | パケット間の待ち時間を `0.5` 秒に設定（0.2未満はroot権限が必要）。 |
| **タイムアウト設定** | `ping -t 2 8.8.8.8` | `ping -w 2000 8.8.8.8` | パケットを失われたと判断するまでのタイムアウトを秒（macOS）またはミリ秒（Windows）で設定。 |

### 詳解：Wireshark 表示フィルタ

キャプチャ*中*にどのパケットをディスクに記録するかを決める**キャプチャフィルタ**と違い、**表示フィルタ**は既にキャプチャ済みのパケットに適用します。数千のパケットを瞬時に検索・非表示・絞り込みできます。

Wireshark はフィルタバーで、とても直感的な色分けされた構文を使います:
* **緑の背景**: 構文が正しい。
* **赤の背景**: 構文にエラーがあり実行されない。

#### 1. よく使う比較演算子と論理演算子

| 演算子 | 別表記 | 意味 | 例 |
| :--- | :--- | :--- | :--- |
| **`==`** | `eq` | 等しい | `ip.addr == 192.168.1.10` |
| **`!=`** | `ne` | 等しくない | `tcp.port != 80` |
| **`&&`** | `and` | 論理AND（両方が真） | `ip.addr == 192.168.1.10 && tcp` |
| **`\|\|`** | `or` | 論理OR（どちらかが真） | `dns or http` |
| **`!`** | `not` | 論理NOT（否定） | `!arp`（すべてのARPを非表示） |
| **`contains`** | *なし* | テキスト文字列を検索 | `http.host contains "example"` |

#### 2. ネットワーク解析の必須フィルタレシピ

| カテゴリ | フィルタ式 | 表示される内容 |
| :--- | :--- | :--- |
| **プロトコルフィルタ** | `http` | 標準的なHTTPパケットのみ表示。 |
| | `dns` | すべてのDNSクエリと応答を表示。 |
| | `icmp` | ping要求/応答などの制御メッセージを表示。 |
| | `arp` | アドレス解決のマッピング要求/応答を表示。 |
| **IPアドレス** | `ip.addr == 192.168.1.10` | このIPが送信元・宛先いずれかのパケットを表示。 |
| | `ip.src == 10.0.0.5` | この機器が送信元のパケットのみ表示。 |
| | `ip.dst == 8.8.8.8` | Google DNS宛のトラフィックのみ表示。 |
| **ポート** | `tcp.port == 80` | 標準HTTPポート80のWebトラフィック（送受信）を表示。 |
| | `udp.port == 53` | 標準DNSポート53のトラフィックを表示。 |
| | `tcp.srcport == 443` | 送出される暗号化HTTPS応答を表示。 |
| **複合フィルタ** | `ip.addr == 192.168.1.10 && tcp.port == 80` | 特定ホストとのWebトラフィックを切り出す。 |
| | `http.request.method == "GET"` | HTTP GETリクエストのみ表示（Web閲覧の追跡に便利）。 |

> [!TIP]
> **右クリックの小技**: フィルタ名が思い出せないときは、**パケット詳細**ペイン（中央）の中でフィールドを見つけ、**右クリック**して **Apply as Filter** → **Selected** を選びます。Wireshark が正しい構文を自動で組み立ててくれます！
