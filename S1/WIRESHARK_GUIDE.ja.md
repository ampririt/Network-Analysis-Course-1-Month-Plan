### Wireshark とは？

**Wireshark** は世界で最も普及している **パケットスニファ**（*パケットアナライザ*／*プロトコルアナライザ*とも呼ばれます）です。無料・オープンソースのツールで、コンピュータ上で動くネットワークプロトコルを「動作中のまま」*見る*ことができます — プロトコル間で実際にやり取りされるメッセージの順序を観察し、各プロトコルフィールドの詳細を掘り下げ、（Webページの読み込みなど）あなたの操作がどうメッセージの送受信を引き起こすかを目で確認できます。

> *「聞いたことは忘れる。見たことは覚える。やったことは理解する。」* — 中国のことわざ

パケットスニファは **受動的（パッシブ）** です。コンピュータ上のアプリやプロトコルが送受信するメッセージを*観察するだけ*で、自分からパケットを送ることはなく、パケットがスニファ宛に明示的に送られることもありません。代わりに、ネットワークインターフェイスを通過するすべてのフレームの**コピー**を受け取ります。

### パケットスニファの仕組み

<p align="center">
  <img src="./img/packet%20sniffer%20structure.png" alt="パケットスニファの構造" width="600"><br>
  <em>図1 — パケットスニファの構造。</em>
</p>

パケットスニファ（上図の破線の四角）は、コンピュータの通常のネットワークソフトウェアに付け加わるもので、**2つの部分**から成ります:

1. **パケットキャプチャライブラリ（`pcap`）** — あるインターフェイス（Ethernet または Wi-Fi）で送受信される**すべてのリンク層フレーム**の*コピー*を受け取ります。HTTP・DNS・TCP・UDP・IP など*すべての*上位層メッセージは最終的にリンク層フレームにカプセル化されるため、全フレームをキャプチャすれば、マシン上の**すべての**プロトコル・アプリが送受信する**すべてのメッセージ**を取得できます。
2. **パケットアナライザ** — キャプチャしたフレームの構造とその中に入れ子になったプロトコルを理解します。各プロトコルフィールド（Ethernet → IP → TCP/UDP → HTTP …）の内容を読める形にデコードして表示します。

これは [セッション1の講義](./README.ja.md#講義) で学んだ **OSI / TCP-IP のカプセル化** にそのまま対応します。キャプチャライブラリが生の **第2層フレーム**を取得し、アナライザが **第3層（IP）**・**第4層（TCP/UDP）**・**第7層（アプリケーション）** を順に剥がして見せてくれます。

### キャプチャ前のチェックリスト

以下の設定で、トラフィックが暗号化されたノイズではなく **読める平文** になるようにします:

- [ ] **VPN をすべてオフにする。** VPN は上位層（HTTP/TCP）の情報を暗号化し、Wireshark から見えなくします。
- [ ] **ブラウザの HTTP/3 と QUIC を無効化する。** 2025年時点で多くのブラウザは既定でこれらを使い、上位層の情報を暗号化します。（ガイド: <https://techysnoop.com/disable-quic-protocol-in-chrome-edge-firefox/>）
- [ ] **`https://` ではなく `http://` のURLを使う。** HTTPS はフレームの中身を暗号化します。
- [ ] **トラフィックをブロック／迂回させうるブラウザのプライバシー／追跡防止機能をオフにする。**
- [ ] **ブラウザのキャッシュと履歴をクリアする** — ページがキャッシュではなく実際にネットワーク越しに取得されるように。

### Wireshark ウィンドウの5つの部分

キャプチャ中のウィンドウは図3のようになります。**5つの主要コンポーネント**があります:

<p align="center">
  <img src="./img/Wireshark%20window%2C%20during%20and%20after%20capture.png" alt="キャプチャ中・後の Wireshark ウィンドウ" width="680"><br>
  <em>図3 — キャプチャ中・後の Wireshark ウィンドウ。</em>
</p>

1. **コマンドメニュー**（上部）— 標準的なプルダウンメニュー。今よく使うのは **File**（キャプチャの保存/開く、終了）と **Capture**（開始/停止）。
2. **表示フィルタ欄** — プロトコル名や式を入力して、見たいパケットだけ表示します（例 `http`）。フィルタは [README の表示フィルタ詳解](./README.ja.md#詳解wireshark-表示フィルタ) で詳しく扱います。
3. **パケット一覧ウィンドウ** — 1パケット1行: Wireshark のパケット**番号**・**時刻**・**送信元/宛先IP**・最上位の**プロトコル**・短い**Info**。列見出しをクリックで並べ替え。
4. **パケット詳細ウィンドウ** — 選択したパケットのプロトコル層のツリー。各層（Frame → Ethernet → IP → TCP/UDP → アプリ）を ▸/▾ で展開/折りたたみ。ここで*カプセル化を層ごとに見られます*。
5. **パケットバイトウィンドウ** — 生フレーム全体を **16進数とASCII** で表示。

> 3・4・5は連動しています。一覧（3）でパケットを選ぶと、その層が詳細（4）に、生バイトがバイト欄（5）に表示されます。

### HTTP について少しだけ（キャプチャするトラフィック）

下のラボではWebページを読み込むので、キャプチャするトラフィックは **HTTP** — Webブラウザとサーバが使うプロトコルです。本質的に HTTP は、ブラウザ（*クライアント*）とWebサーバの間の単純な **リクエスト → レスポンス** の会話です:

<p align="center">
  <img src="./img/Basic%20Application%20Logic%20to%20Get%20a%20Web%20Page.png" alt="ブラウザがサーバにWebページを要求する図" width="560"><br>
  <em>図4 — ブラウザ（「Bob」）がサーバ（「Larry」）にページを要求し、サーバがファイルを返す。</em>
</p>

もう少し詳しく見ると、各メッセージはそれを説明する **HTTPヘッダ** を持ちます。ブラウザは欲しいファイルを指定する **`GET`** リクエストを送り、サーバはステータス（**`200 OK`**）ヘッダに続けてファイルのデータを返します（複数メッセージにまたがることもあります）:

<p align="center">
  <img src="./img/HTTP%20GET%20Request%2C%20HTTP%20Reply%2C%20and%20One%20Data-Only%20Message.png" alt="HTTP GET リクエスト、HTTP レスポンス、データのみメッセージ" width="560"><br>
  <em>図5 — HTTP <code>GET</code> リクエスト、<code>200 OK</code> レスポンス、後続のデータメッセージ。</em>
</p>

Wireshark で探すのはまさにこれらのメッセージです: ブラウザが送る **`GET`** と、サーバが返す **`200 OK`**。この図を頭に置いてキャプチャしてください。

### ラボ1 — Wireshark を試運転する

ツールを学ぶ最良の方法は使うことです。このラボでは実際のHTTPページのダウンロードをキャプチャし、その中の `GET` リクエストを見つけます。

1. **Webブラウザを起動**（シークレット/プライベートモード推奨）。
2. **Wireshark を起動。** 起動画面（図2）が出ます — まだキャプチャは始まっていません。
3. **キャプチャ開始。** アクティブなインターフェイス（例 `Wi-Fi: en0`）をダブルクリック、または **Capture → Options → Start**。ウィンドウがライブパケットで埋まります（図3）。
4. **トラフィックを発生させる。** ブラウザで次のURLを入力して読み込みます:
   ```text
   http://gaia.cs.umass.edu/wireshark-labs/INTRO-wireshark-file1.html
   ```
   ブラウザが `gaia.cs.umass.edu` のHTTPサーバに接続し、HTTPメッセージをやり取りして、1行の「congratulations」ページをダウンロードします。これらのフレーム（とその他多数）がキャプチャされます。
5. **キャプチャ停止。** 赤い四角の **🟥 Stop** ボタン、または **Capture → Stop**。1ページ読み込んだだけなのに*他の*プロトコル（DNS・TCP・TLS・QUIC…）がいくつも現れたことに注目 — 見た目以上に色々起きています。
6. **表示フィルタを適用。** フィルタ欄に `http`（小文字 — Wireshark のプロトコル名は*すべて*小文字）と入力し **Enter / Apply**。HTTPメッセージだけが残ります。フィルタバーが**緑**なら構文が正しい証拠です:

   <p align="center">
     <img src="./img/Screenshot%202026-06-08%20at%209.41.54.png" alt="緑のフィルタバーに入力した http 表示フィルタ" width="680"><br>
     <em>図6 — 緑（有効）のフィルタバーに入力した <code>http</code> 表示フィルタ。</em>
   </p>
7. **HTTP `GET` を見つける。** パケット一覧で、Info に **`GET`** に続けて `gaia.cs.umass.edu` のURLが出ている行を探してクリック。
8. **層を探検する（OSIの実践）。** **パケット詳細**ペインで **Frame**・**Ethernet II**・**Internet Protocol**・**Transmission Control Protocol** を折りたたみ（▸）、**Hypertext Transfer Protocol** を展開（▾）します。生のHTTP `GET` リクエスト — `Host`・`User-Agent`・`Accept` ヘッダ — が読めます。

   <p align="center">
     <img src="./img/Screenshot%202026-06-08%20at%2011.07.29.png" alt="HTTP GET リクエストを選択し各層を展開した状態" width="700"><br>
     <em>図7 — 選択した <code>GET</code> パケット。宛先IP <code>128.119.245.12</code>（gaia）、<strong>Dest Port: 80</strong>、展開したHTTP層の中の <code>Host</code> / <code>User-Agent</code> ヘッダに注目。</em>
   </p>
9. **サーバの応答を見る。** サーバから戻る **`HTTP/1.1 200 OK`** パケットをクリック。**Hypertext Transfer Protocol** 層を展開し、**パケットバイト**ペインを見ると、サーバが送った実際のページ本文が読めます。

   <p align="center">
     <img src="./img/Screenshot%202026-06-08%20at%2010.05.14.png" alt="ページ本文が見える HTTP 200 OK レスポンス" width="700"><br>
     <em>図8 — <code>200 OK</code> レスポンス。バイト欄にページ内容「Congratulations! You've downloaded the first Wireshark lab file!」が見えます。</em>
   </p>
10. **Wireshark を終了。** 🎉 **おめでとうございます — はじめての Wireshark ラボ完了です！**

> 💾 キャプチャを保存するには **File → Export Specified Packets…**（または **File → Save As…**）で `.pcap` / `.pcapng` ファイルとして保存します（例 `my_first_capture.pcap`）。

### ラボ1 設問

自分のキャプチャ（またはライブできない場合はダウンロードしたトレースファイル）から答えてください。**まず自分で考えてから「答えを表示」をクリック。**

**Q1.** トレースの *Protocol* 列に **どのプロトコルが現れますか**: TCP, QUIC, HTTP, DNS, UDP, TLSv1.2？

<details>
<summary>💡 答えを表示</summary>

6つすべて現れ得ます。ページの読み込み自体は **DNS**（`gaia.cs.umass.edu` の解決）の後、**TCP** + **HTTP**（取得）を使います。残りの **UDP**・**QUIC**・**TLSv1.2** は、キャプチャ中に他のアプリが出す*バックグラウンド*トラフィックです。（どれが出るかはマシン次第。このページに必須の3つは **DNS・TCP・HTTP**。）
</details>

**Q2.** HTTP `GET` の送信から HTTP `200 OK` の受信まで **どれくらい** かかりましたか？ *(Time 列は既定でトレース開始からの秒数。時刻表示にするには **View → Time Display Format → Time-of-day**。)*

<details>
<summary>💡 答えを表示</summary>

`200 OK` パケットの **Time** から `GET` パケットの **Time** を引きます。これはサーバへの往復時間で、たいてい1秒未満（多くは **約0.02〜0.3秒**、サーバまでの距離やネットワーク状況による）。例: OK が `1.452 s`、GET が `1.410 s` → **0.042秒（42ms）**。
</details>

**Q3.** **`gaia.cs.umass.edu`（別名 `www-net.cs.umass.edu`）のIPアドレスは？** **そして自分のコンピュータ**（`GET` を送ったホスト）のIPは？

<details>
<summary>💡 答えを表示</summary>

`gaia.cs.umass.edu` は **`128.119.245.12`** に解決されます — `GET` パケットの **宛先IP**（または `200 OK` の **送信元IP**）として読めます。**図7** のIPv4層を参照。**自分のコンピュータのIP** は `GET` の **送信元IP** — プライベートアドレス（図7では例として `172.20.2.203`、自分の環境では `192.168.x.x` や `10.x.x.x` のこともあります）。
</details>

**Q4.** HTTP `GET` を運ぶTCPパケットを選びます。**どの種類のWebブラウザ**がリクエストを出しましたか？ *(展開したHTTP部の `User-Agent:` の後の値を読む。)*

<details>
<summary>💡 答えを表示</summary>

使ったブラウザ次第 — **`User-Agent:`** ヘッダがそれを示します。**図7** では `…AppleWebKit/537.36 (KHTML, like Gecko) Chrome/140…`、つまり **Chrome**。（環境により `Firefox/…`・`Safari/…`・`Edg/…` のことも。）これがまさに、Webサーバがどのブラウザから接続されているか知る仕組みです。
</details>

**Q5.** 同じパケットの **Transmission Control Protocol** 部を展開します。HTTPリクエストが送られた **宛先ポート番号**（`Dest Port:` の後）は？

<details>
<summary>💡 答えを表示</summary>

**`80`** — HTTPのウェルノウンポート。TCP層に **Dest Port: 80** と表示されます（**図7**参照）。（ブラウザ自身の**送信元**ポートはランダムな高番号、例 `54321`。）
</details>

**Q6.** **2つのHTTPメッセージ（`GET` と `200 OK`）を印刷する**: **File → Print** で **"Selected Packet Only"** と **"Print as displayed"** を選び **OK**。

<details>
<summary>💡 答えを表示</summary>

これは調べる事実ではなく手を動かす手順です。`GET` パケットを選び **File → Print**、**"Selected Packet Only"** と **"Print as displayed"** にチェックして **OK**。`200 OK` でも同様に。ヒント: PDFに印刷すれば電子提出できます。
</details>

### アクティビティ2 — ディープパケットインスペクション（層ごとに）

1つのパケットは実は**入れ子のヘッダ**の集まりです — スタックの各層が、上の層のデータを包みます（**カプセル化**）。Wireshark の**パケット詳細ペイン**は、その包みを**一層ずつ**剥がせます。これが*ディープパケットインスペクション*です。ここでは1つの **TCP SYN** パケット（TCPハンドシェイクの最初のパケット）を、最も外側から内側へ分解します。

> 一覧でパケットを選び、詳細ペインの ▸ をクリックして各層を展開します。上から下へ — **Frame → Ethernet II → IP → TCP** — これはワイヤ上で外側→内側の順でもあります。

1. **Frame — Wireshark のキャプチャメタデータ（本物のヘッダではない）。** 最外項目はワイヤを流れたプロトコルではなく、Wireshark がキャプチャ*について*記録したもの: **フレーム番号**・**到着時刻**・全体**長**（74バイト）・検出したプロトコルスタック（`eth:ethertype:ip:tcp`）。Wireshark がパケットに貼るラベルのようなものです。

   <p align="center">
     <img src="./img/Deep%20Packet%20Inspection%20(Layer%20by%20Layer)/Frame%20Network%20Packet%20Information.webp" alt="Wireshark で展開した Frame 層" width="720"><br>
     <em>図9 — <strong>Frame</strong> 層: キャプチャのメタデータ（番号・時刻・長さ・検出プロトコル <code>eth:ethertype:ip:tcp</code>）。</em>
   </p>
2. **Ethernet II — 第2層（データリンク）。** フレームヘッダは*このホップ*の**送信元/宛先MAC**アドレスと、中身が何かを示す **Type** フィールド（`IPv4, 0x0800`）を持ちます。これがホップごとに書き換えられる唯一の層です。

   <p align="center">
     <img src="./img/Deep%20Packet%20Inspection%20(Layer%20by%20Layer)/Ethernet%20II%20Network%20Packet%20Information%20.webp" alt="Wireshark で展開した Ethernet II 層" width="720"><br>
     <em>図10 — <strong>Ethernet II</strong> ヘッダ（第2層）: 送信元/宛先 <strong>MAC</strong> と Type <code>IPv4 (0x0800)</code>。</em>
   </p>
3. **Internet Protocol Version 4 — 第3層（ネットワーク）。** IPヘッダは**エンドツーエンド**の送信元（`172.21.224.2`）と宛先（`142.250.1.139`）の **IP** アドレス、**TTL**（`64`）、**Protocol** フィールド（`TCP (6)` — 次の入れ子）、全長、ヘッダチェックサムを持ちます。

   <p align="center">
     <img src="./img/Deep%20Packet%20Inspection%20(Layer%20by%20Layer)/Internet%20Protocol%20Packet%20Information%20.webp" alt="Wireshark で展開した IPv4 層" width="720"><br>
     <em>図11 — <strong>IPv4</strong> ヘッダ（第3層）: 送信元/宛先 <strong>IP</strong>、<strong>TTL 64</strong>、Protocol <code>TCP (6)</code>。</em>
   </p>
4. **Transmission Control Protocol — 第4層（トランスポート）。** TCPヘッダは **送信元ポート**（`49652`）と**宛先ポート**（`80` = HTTP）、**シーケンス/確認応答**番号、**Flags** — ここでは `0x002 (SYN)`、これが **3ウェイハンドシェイクの最初のパケット**であることを示します。Options には MSS・SACK・タイムスタンプ・ウィンドウスケールなど。

   <p align="center">
     <img src="./img/Deep%20Packet%20Inspection%20(Layer%20by%20Layer)/TCP%20Packet%20Information.webp" alt="Wireshark で展開した TCP 層" width="720"><br>
     <em>図12 — <strong>TCP</strong> ヘッダ（第4層）: 送信元/宛先 <strong>ポート</strong>、シーケンス番号、Flags <code>0x002 (SYN)</code>。</em>
   </p>

4層をまとめると、1つのパケットの中にスタック全体が入っており、各層がどのアドレスを使うかが分かります:

| 詳細ペインの層 | OSI層 | 主なフィールド | アドレスの種類 |
| :--- | :--- | :--- | :--- |
| **Frame** | —（Wireshark メタデータ） | フレーム番号・時刻・長さ・プロトコル一覧 | — |
| **Ethernet II** | L2 — データリンク | 送信元/宛先MAC、Type | **MAC**（ホップごと） |
| **Internet Protocol v4** | L3 — ネットワーク | 送信元/宛先IP、TTL、Protocol | **IP**（エンドツーエンド） |
| **Transmission Control Protocol** | L4 — トランスポート | 送信元/宛先ポート、Seq/Ack、Flags | **ポート** |
| *(HTTP がある場合)* | L7 — アプリケーション | リクエスト/データ本体 | — |

> これは **逆向きのカプセル化** です: アプリのデータが TCP → IP → Ethernet と包まれてワイヤに乗り、Wireshark が外側から順に解いて見せます。各層の **Type/Protocol** フィールドが、次の層をどう解釈するかを受信側に教える「パンくず」です（`Ethernet Type 0x0800 → IPv4`、`IP Protocol 6 → TCP`）。

#### アクティビティ2 設問

**まず自分で考えてから「答えを表示」をクリック。**

**Q1.** **Frame** 部はツリーの一番上にありますが、ネットワークを流れたものではありません。これは何ですか？

<details>
<summary>💡 答えを表示</summary>

キャプチャしたパケットについての **Wireshark 自身のメタデータ** です — フレーム番号・到着時刻・キャプチャ長・検出したプロトコル一覧。キャプチャツールが付加したもので、どのホストも送っていないため「ワイヤ上のバイト」はありません。
</details>

**Q2.** **MAC** アドレスを持つのはどの層で、**IP** アドレスを持つのはどの層？ ホップごとに変わるのはどちら？

<details>
<summary>💡 答えを表示</summary>

**Ethernet II（第2層）** が **MAC**、**IPv4（第3層）** が **IP**。**MAC** は**ルータのホップごと**に（リンク単位で）書き換えられ、**IP** はエンドツーエンドで同じ — 「IPは不変、MACはホップごとに変わる」の原則です。
</details>

**Q3.** TCP の **Flags** は `0x002 (SYN)` です。このパケットについて何が分かりますか？

<details>
<summary>💡 答えを表示</summary>

**TCP 3ウェイハンドシェイクの最初のパケット** — クライアントが接続を開く合図（「話せますか？」）です。完全な `SYN → SYN-ACK → ACK` のやり取りは **[セッション2](../S2/WIRESHARK_GUIDE.ja.md)** でキャプチャします。
</details>

**Q4.** IPv4 ヘッダの **Protocol** フィールドは `TCP (6)` です。なぜこのフィールドが Wireshark（と受信ホスト）にとって重要？

<details>
<summary>💡 答えを表示</summary>

IPペイロードの**中身が何か**を受信側に伝えるので、次の層を **TCP**（UDPやICMPではなく）として解釈できます。各層にこの「中身は何か」フィールドがあり、Ethernet の **Type `0x0800`** も同様に「中はIPv4」と示しました。これがデコーダがスタックを下っていく仕組みです。
</details>

### アクティビティ3 — ARP：このIPのMACは誰？

ローカルネットワークでパケットを送るたびに、コンピュータは次の機器（LANのホストやゲートウェイ）の **MACアドレス** を必要とします。*IP* は既に分かっているので、こう尋ねます: *「このIPは誰？ MACを教えて。」* この問いと答えが **ARP（Address Resolution Protocol）** — **第3層（IP）** と **第2層（MAC）** をつなぐ接着剤です。このアクティビティで ARP を可視化し、裏で何が起きているかを説明します。

#### ARP の仕組み（メカニズム）

この例では、**あなたのコンピュータは `172.20.2.203`** で、**`172.20.0.1`**（ゲートウェイ）に到達したいが、その *IP* しか知らず *MAC* は知りません。コンピュータは実は**2つの**アドレスを持ち、それぞれ別の層にあります:

| | アドレス | 例（このラボ） | 層 | 範囲 | 割り当て元 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **論理** | IP | `172.20.2.203`（自分）・`172.20.0.1`（相手） | 第3層 | インターネット全体でエンドツーエンド | DHCP / 設定 |
| **物理** | MAC | `a4:83:e7:1c:09:5b` | 第2層 | 1つのローカルリンクのみ | NICに焼き込み |

ルーティングやアプリは **IP** アドレスで動きますが、フレームは物理的に **MAC** アドレス宛にしか配送できません。ARP は「IP X に届けたい」を「MAC Y にフレームを送る」に変換するルックアップです。全サイクルは次の通り:

1. **まずキャッシュを見る。** OS は最近学習した IP→MAC の対応を **ARPキャッシュ**（`arp -a`）に保持します。既にあれば ARP トラフィックは不要。
2. **セグメント全体に尋ねる（ブロードキャスト）。** ミスなら、ホストは **ARP リクエスト**を `ff:ff:ff:ff:ff:ff` にブロードキャストします — ローカルリンクの*すべて*の機器が受信: *「`172.20.0.1` は誰？ `172.20.2.203` に教えて。」*
3. **持ち主だけが答える（ユニキャスト）。** そのIPの持ち主（`172.20.0.1`）が **ARP リプライ**をあなたに直接返します: *「`172.20.0.1` は `a4:83:e7:1c:09:5b` です。」* 他は沈黙。
4. **キャッシュして再利用。** 両者が対応を保存し、再度尋ねずに済みます。エントリは数分（または機器が静かになると）で**期限切れ**になり、テーブルを新鮮に保ちます。

<p align="center">
  <img src="./img/arp/How%20an%20ARP%20request%20works.gif" alt="ARP リクエストのブロードキャストとユニキャスト応答のアニメーション" width="640"><br>
  <em>図13 — ARP が IP を MAC に解決する流れ: リクエストは全員にフラッディングされ、持ち主だけが応答。</em>
</p>

> [!NOTE]
> **ARP はローカルリンクを出ません。** リクエストはブロードキャストで、**ルータはブロードキャストを転送しない**ため、ARP は単一のサブネット／ブロードキャストドメインに限定されます。これが下のサブネットをまたぐ設問の鍵です。（ARP は IPv4 専用。IPv6 は **NDP（Neighbor Discovery Protocol）** が同じ役割を担います。）

> [!TIP]
> **Gratuitous ARP（無償ARP）** は、ホストが**自分自身**のIPについて送る*要求されていない*リプライ（target IP = sender IP）です。近隣のキャッシュを先に埋める・新機器を告知する・**IP競合を検出する**のに使われます（誰かが答えたら2台が同じIP）。ARP には**認証がない**ため、**ARP スプーフィング**（セッション3で扱う）でも悪用されます — どのホストもどのIPでも名乗れます。

#### ARP のやり取りをキャプチャする

1. **ARPキャッシュを見る。** ターミナルで `arp -a` を実行し、既知の IP→MAC 対応を確認。新しい ARP トラフィックを起こすには、まずフラッシュ:
   ```sh
   # macOS / Linux
   sudo arp -d -a
   # Windows（管理者）
   arp -d *
   ```
2. **キャプチャしてフィルタ。** キャプチャを開始し、フィルタ欄に **`arp`** と入力して **Enter**。
3. **ARP のやり取りを発生させる。** ゲートウェイやLAN上の別機器に `ping`（ここでは `ping 172.20.0.1`）。pingを送る前にそのIPのMACを解決する必要があるので、最初のICMP echo の*直前*に ARP の要求/応答が現れます。
4. **リクエストを調べる。** **ブロードキャスト**の ARP パケットをクリックし **Address Resolution Protocol** を展開。**Opcode `request (1)`**、**Ethernet 宛先 = `ff:ff:ff:ff:ff:ff`**（ブロードキャスト）、自分の **Sender MAC/IP**、そして肝心の **Target MAC = `00:00:00:00:00:00`**（埋めるべき空欄）に注目。Info は *"Who has 172.20.0.1? Tell 172.20.2.203."*。

   <p align="center">
     <img src="./img/arp/arp_broadcast_request.png" alt="ARP リクエスト: ブロードキャスト宛先とゼロの Target MAC" width="720"><br>
     <em>図14 — ARP <strong>リクエスト</strong>。ブロードキャストMAC宛で、Sender は*自分*のMAC/IP、**Target MAC は全ゼロ** — まさにそれを尋ねているから。</em>
   </p>
5. **リプライを調べる。** 対応する**ユニキャスト**の ARP パケットをクリック。**Opcode = `reply (2)`**、**Sender MAC** が欲しかった答え（`172.20.0.1` の持ち主）で、**Target MAC/IP** は*自分*のアドレス。Info は *"172.20.0.1 is at `<欲しいMAC>`."*。

   <p align="center">
     <img src="./img/arp/arp_response.png" alt="ARP リプライ: 欲しい MAC が埋まり自分宛にユニキャスト" width="720"><br>
     <em>図15 — ARP <strong>リプライ</strong>。**欲しいMAC** が Sender MAC 欄を埋め、*自分*のMAC宛に直接（ユニキャスト）返ります。</em>
   </p>
6. **キャッシュを確認。** もう一度 `arp -a` を実行すると、ping したIPに対しMACが保存され、期限切れまで再度尋ねずに済みます。

#### ARP パケットのフィールドを読む

**Address Resolution Protocol** を展開すると、どのARPメッセージ（要求/応答）も次のフィールドを持ちます。コツは2つのパケットを見比べることです（**図14** 要求 vs **図15** 応答）:

| フィールド | **リクエスト**で | **リプライ**で | 意味 |
| :--- | :--- | :--- | :--- |
| **Hardware type** | `Ethernet (1)` | `Ethernet (1)` | 使う第2層メディア |
| **Protocol type** | `IPv4 (0x0800)` | `IPv4 (0x0800)` | 解決する第3層アドレス |
| **Opcode** | `request (1)` | `reply (2)` | 問い vs 答え — 形式は同じ |
| **Sender MAC** | 自分のMAC | **答える側**のMAC ← *目当て* | このARPを送ったのは誰 |
| **Sender IP** | `172.20.2.203`（自分） | `172.20.0.1`（尋ねた相手） | 送信側のIP |
| **Target MAC** | `00:00:00:00:00:00`（*未知！*） | 自分のMAC | 埋められるフィールド |
| **Target IP** | `172.20.0.1`（尋ねている相手） | `172.20.2.203`（自分） | このメッセージの対象 |

> リクエストの全ゼロの **Target MAC**（図14）は視覚的な「空欄を埋めよ」です — この空欄こそ ARP が存在する全理由。リプライ（図15）はそこ（と Sender MAC）が埋まって届き、それがコンピュータがキャッシュする IP→MAC 対応です。

#### アクティビティ3 設問

**まず自分で考えてから「答えを表示」をクリック。**

**Q1.** ARP *リクエスト*はどの **宛先MAC** を使い、それはなぜ？

<details>
<summary>💡 答えを表示</summary>

**ブロードキャスト**アドレス `ff:ff:ff:ff:ff:ff`。送信側はまだ相手のMACを知らない（だから尋ねている）ので、ローカルセグメントの**全**機器に問いを送る必要があります — そのIPの持ち主だけが答えます。
</details>

**Q2.** ARP リクエストとリプライの **Opcode** 値は？

<details>
<summary>💡 答えを表示</summary>

**Request = `1`**、**Reply = `2`**。形式は同じで、問いと答えを区別するのが opcode です。
</details>

**Q3.** ARP はどの **2つの層** を橋渡しし、なぜそれが必要？

<details>
<summary>💡 答えを表示</summary>

ARP は **第3層のIPアドレス** を **第2層のMACアドレス** に対応づけます。パケットはIPで宛先指定しますが、実際のローカル配送（Ethernet/Wi-Fi フレーム）にはMACが必要 — ARP が各ローカルリンクでその隙間を埋めます。
</details>

**Q4.** **別サブネット**のホストに `ping` したとき、ARP が解決するのは誰のMAC — 相手ホストの？ それとも別の？

<details>
<summary>💡 答えを表示</summary>

**デフォルトゲートウェイ**のMACで、相手ホストのではありません。ARP は**ローカルリンク**でしか効かず、リモートホストのMACは無意味です。だからコンピュータはゲートウェイをARPし、そこにパケットを渡す — ルータが先へ転送します（そして次のリンクでARP）。ルーティングの「IPは不変、MACはホップごとに変わる」と同じ考えです。
</details>

**Q5.** ARP **リクエスト**の **Target MAC** は全ゼロ（`00:00:00:00:00:00`）です。なぜ？

<details>
<summary>💡 答えを表示</summary>

相手のMACこそ**送信側がまだ知らないもの** — リクエストが埋めようとしている空欄だからです。送信側は*相手のIP*は知っているのでそれを記入し、Target MAC は空のまま。リプライでそのフィールド（と Sender MAC）が埋まって戻ります。
</details>

**Q6.** ARP には**認証がなく**、どのホストもどのIPでも答えられます。これがどんな攻撃を可能にし、どこで扱いますか？

<details>
<summary>💡 答えを表示</summary>

**ARP スプーフィング／ポイズニング** — 攻撃者がゲートウェイのIPは*自分*のMACだと偽の（しばしば無償の）ARPリプライを送り、被害者のトラフィックを攻撃者へ向けさせます（中間者攻撃）。防御はスイッチの **Dynamic ARP Inspection（DAI）** で、信頼された IP↔MAC 対応に合わない ARP リプライを破棄します。
</details>
