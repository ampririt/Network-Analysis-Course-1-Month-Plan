school ネットワーク基礎知識

ネットワークの重要コンセプトを、カテゴリ別に分かりやすく整理しました。

location_on

アドレスと名前解決

ネットワーク上で「誰に届けるか」を決める仕組み

location_on IPアドレス

ネットワーク上の機器の「住所」（例：192.168.1.1）
  • 構造：32ビットを「ネットワーク部」と「ホスト部」に分割
  • ネットワーク部：どの「町」に属するか → ルータが判断
  • ホスト部：その「町」の中の個別の「家」を識別
  • プライベートIP：10.x / 172.16〜31.x / 192.168.x は社内用
住所の「東京都新宿区（ネットワーク部）＋ 1丁目2番3号（ホスト部）」と同じ考え方

badge MACアドレス

機器に製造時から付いた48ビットの「名札」（例：00:1A:2B:3C:4D:5E）
  • 前半24bit（OUI）：メーカーの識別番号（IEEE管理）
  • 後半24bit：メーカーが割り当てるシリアル番号
  • 使用場所：L2（データリンク層）で同一ネットワーク内の通信に利用
  • FF:FF:FF:FF:FF:FF = ブロードキャスト（全員宛て）
IPアドレスが「住所」なら、MACアドレスは変更できない「マイナンバー」のようなもの

dns DNS (Domain Name System)

ドメイン名（www.google.com）→ IPアドレスに変換する「電話帳」
  • 仕組み：ルートDNS → TLD(.com等) → 権威DNS の順に階層検索
  • キャッシュ：一度調べた結果を保存し、2回目は即回答
  • 通信：UDPポート53 を使用（高速応答のため）
  • レコード：A=IPv4 / AAAA=IPv6 / MX=メール / CNAME=別名
スマホの電話帳で「田中さん」と入力すると「090-xxxx」に変換されるのと同じ

swap_horiz ARP (Address Resolution Protocol)

IPアドレスからMACアドレスを見つける「翻訳プロトコル」
  • Request：「このIPのMACアドレスは？」と全員にブロードキャスト
  • Reply：該当する機器だけがユニキャストでMACを返す
  • ARPテーブル：結果をキャッシュして次回はARP不要
  • 位置：L2とL3の橋渡し役
教室で「席番号5番の人、名前教えて！」と全員に聞いて、該当者だけが答えるイメージ
sync_alt

プロトコルと通信

データをどうやって正確に届けるかのルール

lan TCP / UDP (Transmission Control / User Datagram Protocol)

L4の2大プロトコル — 「信頼性」か「速度」かの選択
  • TCP：3-wayハンドシェイクで確実に届ける（Web, メール, FTP）
  • UDP：確認なしで高速に送る（DNS, 動画, VoIP, ゲーム）
  • ポート番号：HTTP=80 / HTTPS=443 / SSH=22 / DNS=53
  • TCP接続：SYN → SYN+ACK → ACK の3ステップで開始
TCP=書留郵便（届いたか確認）、UDP=ハガキ投函（届いたかは不明だが速い）

podcasts ICMP (Internet Control Message Protocol)

ネットワークの「健康診断」用プロトコル（ping / tracerouteで使用）
  • Echo Request/Reply：相手が通信できる状態か確認（ping）
  • Destination Unreachable：宛先に到達不可の通知
  • Time Exceeded：TTL=0 で破棄された場合の通知（traceroute活用）
  • 動作層：L3（ネットワーク層）でIPの補助的役割
お医者さんが「もしもし、聞こえますか？」と確認するような診断ツール

view_column ヘッダ

パケットの先頭につく「荷札」— 各層で異なる制御情報を追加
  • Ethernet（14B）：宛先/送信元MAC + タイプ
  • IP（20B〜）：宛先/送信元IP + TTL + プロトコル番号
  • TCP（20B〜）：ポート番号 + シーケンス番号 + フラグ
  • 受信時：各層でヘッダを読んで処理を決定 → ヘッダ除去
宅配便の送り状（住所=IP）、中の封筒の宛名（名前=MAC）、内容物リスト（TCP）

wifi サブネット

大きなネットワークを小さく分割して効率的に管理する仕組み
  • サブネットマスク：ネットワーク部とホスト部の境界線を定義
  • CIDR表記：/24 = 255.255.255.0（上位24bitがネットワーク部）
  • ホスト数：2^(ホスト部ビット数) - 2（先頭と末尾は予約）
  • 目的：部署ごとに分割してブロードキャスト範囲を制限
大きなオフィスビルを「1階=営業」「2階=開発」と部署で分けるイメージ
router

ネットワーク機器

データを正しい場所に届けるための装置たち

router ルータ

異なるネットワーク間をつなぐ「道案内人」（L3動作）
  • ルーティング：宛先IPを見てルーティングテーブルで転送先を決定
  • ブロードキャスト分割：ネットワークを分けてトラフィックを制限
  • 各ポート：異なるIPアドレスとMACアドレスを持つ
  • デフォルトGW：外部ネットワークへの「出口」となるルータ
道路の交差点にいる交通整理員 — 行き先を見て正しい道を案内

device_hub スイッチ

同一ネットワーク内のデータ転送を管理する「交通整理員」（L2動作）
  • MACテーブル（CAM）：学習したMACで宛先ポートだけに転送
  • コリジョンドメイン分割：全二重通信を実現
  • 速度：FastEthernet(100Mbps) / GigabitEthernet(1Gbps)
  • VLAN：論理的にネットワークを分割可能
郵便局の仕分け作業員 — 宛名(MAC)を見て正しいポストに入れる

security ファイアウォール

不正アクセスを防ぐ「門番」— 許可されたトラフィックだけ通す
  • パケットフィルタリング：IP/ポートに基づくルールで制御
  • ステートフル検査：通信の状態を追跡して判断
  • IDS：侵入を検知して管理者に通知
  • IPS：侵入を検知して自動で遮断
建物の受付 — 許可証(ルール)を持つ人だけ入館させる

cable TTL (Time To Live)

パケットの「寿命」— ルータ通過ごとに-1、0で破棄
  • 目的：パケットが無限に回り続けるのを防止
  • 初期値：Windows=128 / Linux=64
  • TTL=0：ICMP Time Exceeded が送信元に返される
  • traceroute：TTLを1,2,3...と増やして各ルータを特定
リレーの走者が「最大10人まで」とルール付き — 10人目で走るのをやめる

lightbulb カプセル化のポイント

送信時：上位層→下位層の順にヘッダを追加（Data → +TCP → +IP → +Eth+FCS）
受信時：下位層→上位層の順にヘッダを除去（Eth除去 → IP除去 → TCP除去 → Data取得）
各層は独立：自分の層のヘッダだけ読み、上位層の内容には関知しない（レイヤー独立の原則）

「ハンドシェイク開始」ボタンでシミュレーションを開始します。TCPが通信を始める前に行う3段階の確認手順をアニメーションで見てみましょう。

arrow_forward Step 1: SYN（接続要求）

クライアントが SYN フラグを立てたパケットを送信。
「通信したいです！」という合図。初期シーケンス番号(ISN)をランダムに生成して送る。
クライアントの状態：CLOSED → SYN_SENT

swap_horiz Step 2: SYN+ACK（応答）

サーバが SYN+ACK で応答。
「了解！私も通信OK！」という意味。サーバも自分のISNを送り、クライアントのseq+1をackとして返す。
サーバの状態：LISTEN → SYN_RECEIVED

check_circle Step 3: ACK（確認完了）

クライアントが ACK を返す。
「確認しました！通信開始！」。サーバのseq+1をackとして送る。
両方の状態が ESTABLISHED（接続確立）になり、データ通信が可能になる。

close 切断（4-Way Handshake）

通信終了時は4段階：
FIN → ACK → FIN → ACK
双方が「もう送るデータはない」と確認し合ってから接続を閉じる（丁寧な終了）。

view_column パケットヘッダ構造の詳細

各プロトコルのヘッダフィールドを視覚的に表示。各フィールドにマウスを乗せると詳しい説明が表示されます。

settings_ethernet Ethernet ヘッダ（L2 データリンク層）— 14 バイト
宛先MAC (6B)
受信側のMACアドレス（48bit）。スイッチはこのアドレスを見てどのポートに転送するか決定する。ブロードキャスト時は FF:FF:FF:FF:FF:FF。
送信元MAC (6B)
送信側のMACアドレス（48bit）。スイッチはこのアドレスをMACテーブル（CAMテーブル）に学習し、送信元のポートを記録する。
タイプ (2B)
上位プロトコルの種類を示す。0x0800=IPv4、0x0806=ARP、0x86DD=IPv6。受信側はこの値で次にどのプロトコルで処理するか判断する。
役割：同一ネットワーク内（L2）でフレームを正しい機器に届ける。末尾にはFCS（Frame Check Sequence, 4B）が付き、CRC32で誤り検出を行う。
language IP (Internet Protocol) ヘッダ（L3 ネットワーク層）— 20 バイト〜
Ver (4bit)
IPのバージョン。IPv4=4、IPv6=6。現在主流はIPv4だが、アドレス枯渇によりIPv6への移行が進行中。
IHL (4bit)
ヘッダ長（Internet Header Length）。4バイト単位で表現。通常は5（=20バイト）。オプション付きは6以上。
ToS/DSCP (8bit)
Type of Service / DSCP (Differentiated Services Code Point)。パケットの優先度やQoS（Quality of Service）を制御。VoIPなどリアルタイム通信で重要。
全長 (16bit)
IPヘッダ＋データを含むパケット全体のバイト数。最大65,535バイト。ただしMTU（通常1500B）で制限され、超過時はフラグメント化される。
TTL (8bit)
Time To Live — パケットの寿命。ルータ通過ごとに-1され、0になると破棄される。ループ防止が目的。Windows初期値=128、Linux=64。tracerouteで活用。
プロトコル (8bit)
上位層のプロトコル番号。6=TCP、17=UDP、1=ICMP。受信側はこの番号を見て、ペイロードをどのプロトコルに渡すか判断する。
チェックサム (16bit)
IPヘッダのみの誤り検出用。データ部分は含まない（データの誤り検出はTCP/UDPのチェックサムが担当）。ルータはTTL変更のたびに再計算する。
送信元IP (32bit)
パケットを送信した機器のIPアドレス。32bit=4オクテット（例：192.168.1.10）。NATではここがグローバルIPに変換される。
宛先IP (32bit)
パケットの最終的な宛先のIPアドレス。ルータはこのアドレスをルーティングテーブルと照合して転送先を決定する。
役割：異なるネットワーク間（L3）でパケットを宛先まで届ける。ルータはIPヘッダの宛先IPを読み、ルーティングテーブルで次の転送先を決定する。
sync_alt TCP (Transmission Control Protocol) ヘッダ（L4 トランスポート層）— 20 バイト〜
送信元ポート (16bit)
送信側アプリケーションの識別番号（0〜65535）。通常、クライアント側は49152〜65535のエフェメラルポートをOSがランダムに割り当てる。
宛先ポート (16bit)
受信側のアプリケーション識別。HTTP=80, HTTPS=443, SSH=22, FTP=21, SMTP=25, DNS=53。Well-known Ports: 0〜1023。
シーケンス番号 (32bit)
送信データのバイト単位の位置番号。受信側がデータを正しい順序に並べ替えるために使用。3-wayハンドシェイクでISN（Initial Sequence Number）をランダムに決定。
確認応答番号 (32bit)
次に受信したいバイトの番号（ACK番号）。「ここまで正常に受信した」という意味。ACKフラグが1の時のみ有効。
フラグ (6bit)
SYN=接続要求, ACK=確認応答, FIN=切断要求, RST=強制切断, PSH=即時処理要求, URG=緊急データ。3-wayハンドシェイクではSYN→SYN+ACK→ACKの順に使用。
ウィンドウ (16bit)
受信バッファの空き容量（バイト数）。フロー制御で「一度に受け取れるデータ量」を通知。相手はこの値を超えて送信しない。
チェックサム (16bit)
ヘッダ＋データ全体の誤り検出。擬似ヘッダ（送信元IP, 宛先IP, プロトコル番号）も含めて計算。破損検出時は再送を要求する。
役割：信頼性のある通信を保証。3-wayハンドシェイクで接続確立、シーケンス番号で順序保証、ACKで到達確認、ウィンドウでフロー制御、再送で損失回復。
speed UDP (User Datagram Protocol) ヘッダ（L4 トランスポート層）— 8 バイト固定
送信元ポート (16bit)
送信側アプリケーションの識別番号。省略可能（0にセット可）。DNS応答の返送先などに使用。
宛先ポート (16bit)
受信側のアプリケーション識別。DNS=53, DHCP=67/68, SNMP=161, RTP(音声/動画)=動的。
データ長 (16bit)
UDPヘッダ(8B)+データの合計バイト数。最小値は8（ヘッダのみ）。TCPと違いシーケンス番号やACKがないためシンプル。
チェックサム (16bit)
誤り検出用。IPv4では省略可能（0にセット）。IPv6では必須。UDPはエラー検出のみで、再送や順序保証は行わない。
役割：コネクションレスで高速な通信。ハンドシェイク不要、順序保証なし、再送なし。ヘッダは僅か8バイト（TCPの20B+と比較して軽量）。DNS, 動画ストリーミング, VoIP, オンラインゲームなどリアルタイム性を重視する通信で使用。
podcasts ICMP (Internet Control Message Protocol) ヘッダ（L3補助）— 8 バイト
タイプ (8bit)
メッセージの種類。0=Echo Reply(ping応答), 3=Destination Unreachable(宛先不達), 8=Echo Request(ping送信), 11=Time Exceeded(TTL超過/traceroute)。
コード (8bit)
タイプの詳細分類。例：タイプ3のコード0=Network Unreachable, コード1=Host Unreachable, コード3=Port Unreachable。
チェックサム (16bit)
ICMPメッセージ全体の誤り検出。ヘッダとデータの両方をカバーする。
識別子 (16bit)
pingプロセスの識別番号。同じPCから複数のpingを同時実行した場合に、どのプロセスへの応答か区別するために使用。
シーケンス番号 (16bit)
パケットの連番。ping送信ごとにインクリメントされ、応答の対応付けやパケットロスの検出に使用する。
役割：ネットワークの診断と制御メッセージ。データ転送には使わず、pingで疎通確認、tracerouteで経路追跡、エラー通知（宛先不達/TTL超過）を行う。TCPやUDPとは異なりポート番号を持たない。

compare_arrows TCP vs UDP ヘッダの比較

TCP（20バイト〜）：シーケンス番号、ACK番号、フラグ、ウィンドウサイズなど多くのフィールドで信頼性を確保。オーバーヘッドが大きい。
UDP（8バイト固定）：ポート番号、データ長、チェックサムのみのシンプル構造。オーバーヘッドが小さく高速。
選択基準：「データが欠けると困る」→ TCP（Web, メール）、「速度が最優先」→ UDP（動画, 音声, ゲーム）

computer 送信PCの処理

L7：HTTPリクエスト生成
L4：TCPセグメント化（送信元ポート=ランダム、宛先ポート=80/443）
L3：IPヘッダ付与（送信元IP=自分、宛先IP=サーバ）
L2：Ethヘッダ付与（宛先MAC=デフォルトGW）。GWのMAC不明時はARPで解決

device_hub L2スイッチの処理

動作層：L2（データリンク層）のみ
処理：宛先MACアドレスをCAMテーブルで検索し、該当ポートへ転送
学習：送信元MACとポートを記録
特徴：IPアドレスは一切見ない。不明MACの場合はフラッディング（全ポートに転送）

router ルータの処理

動作層：L3（ネットワーク層）
処理：① Ethヘッダを除去 → ② 宛先IPをルーティングテーブルと照合 → ③ TTLを-1 → ④ 新しいEthヘッダ（次ホップのMAC）を付与
重要：IPアドレスは変わらないが、MACアドレスはホップごとに書き換わる

speed なぜUDPが速い？

TCPは通信開始前に3-wayハンドシェイク（3往復）が必要で、データ送信中もACK待ちや再送処理がある。UDPはこれらを全て省略し、ヘッダも8バイトと最小限のため、遅延が極めて小さい。

live_tv UDPが適するケース

動画/音声ストリーミング：1パケット欠けても再送より次のフレームを表示した方が良い。
DNS：1回の問い合わせ＋応答で完了する短い通信。
オンラインゲーム：リアルタイム性が最重要。

warning UDPの注意点

データの損失・重複・順序入れ替わりが起きる可能性がある。アプリケーション側で信頼性を補完する場合もある（例：QUIC、RTPのシーケンス番号）。大量のUDPパケット送信はDoS攻撃に悪用されることもある。

person ユニキャスト通信（Unicast）

定義：1対1の通信。特定の1台の宛先にだけデータを送る。
宛先アドレス：特定のMACアドレス / 特定のIPアドレス
例：Webブラウジング（HTTP）、メール送信（SMTP）、SSH接続
特徴：最も一般的な通信方式。ネットワーク帯域を効率的に使用。スイッチはCAMテーブルで該当ポートのみに転送。

group マルチキャスト通信（Multicast）

定義：1対多（特定グループ）の通信。グループに参加した端末だけが受信する。
宛先アドレス：マルチキャストIPアドレス（224.0.0.0〜239.255.255.255, クラスD）
プロトコル：IGMP（Internet Group Management Protocol）でグループ参加/離脱を管理
例：動画ストリーミング配信、IPTVs、ビデオ会議、OSPFルーティング（224.0.0.5）

campaign ブロードキャスト通信（Broadcast）

定義：1対全の通信。同一ネットワーク内の全端末にデータを送る。
宛先アドレス：MAC: FF:FF:FF:FF:FF:FF / IP: ネットワークのブロードキャストアドレス（例: 192.168.1.255/24）
例：ARP Request、DHCP Discover、NetBIOS名前解決
注意：全端末が処理するため帯域を消費。ルータはブロードキャストを転送しない（ブロードキャストドメインの境界）。

特徴ユニキャストマルチキャストブロードキャスト
通信対象1台のみグループ参加者のみ全端末
宛先MAC特定のMAC01:00:5E:xx:xx:xxFF:FF:FF:FF:FF:FF
宛先IP特定のIP224.0.0.0〜239.255.255.255ネットワークの末尾IP
ルータ通過可能設定により可能不可（ドメイン内のみ）
帯域効率最も効率的効率的（必要先のみ）非効率（全員に送信）
主な用途HTTP, SSH, メール動画配信, IPTVARP, DHCP

lightbulb なぜ3種類が必要？

ユニキャスト：通常の通信（特定の相手とだけやり取り）に最適。帯域を無駄にしない。
マルチキャスト：同じデータを複数人に送る場合、ユニキャストで個別に送ると送信者の負荷が大きい。マルチキャストなら1つのパケットでグループ全員に届く。
ブロードキャスト：相手のアドレスがまだわからない場合に必要（ARP: MACを知りたい、DHCP: IPを取得したい）。ただし多用するとネットワーク全体に負荷がかかる。

lan サブネッティング — ネットワークの分割設計

大きなネットワークを部署や用途ごとに分割する設計手法です。「何ビット目で区切るか」が重要なポイントです。

quiz サブネッティングとは？

1つのネットワークを複数の小さなサブネットに分割すること
  • 目的：部署ごとにネットワークを分離し、セキュリティとトラフィック管理を向上
  • 方法：ホスト部のビットの一部を「サブネット部」として借りる
  • トレードオフ：サブネット数を増やすと、各サブネットのホスト数が減る
  • 利点：ブロードキャストドメインを分割し、不要なトラフィックを削減

calculate サブネット設計の手順

「いくつのサブネットが必要か」→「何ビット借りるか」の順で考える
  • Step 1：必要なサブネット数を確認（例：部署が30個）
  • Step 2：2n ≧ サブネット数 となる最小のnを求める
  • Step 3：ホスト部からnビットを借りてサブネット部に
  • Step 4：残りのホスト部ビットで最大ホスト数を確認
assignment 例題：サブネッティング設計

問題：次の条件でサブネッティングを設計せよ
◽ クラスBの 172.16.0.0/16 を使用
◽ 最低でも 30個 の部署（サブネット）が必要

サブネッティングをしない場合

ネットワークは1つで登録可能なホスト数は 216 − 2 = 65,534台。部署を分割できない。

30個のサブネットを表現するのに何ビット必要か？

22 = 4    23 = 8    24 = 16    25 = 32 ≧ 30 ✓

元のホスト部16ビットから5ビット借りると、残りは 11ビット

swap_calls NAT / PAT — アドレス変換

NAT（Network Address Translation）はプライベートIPアドレスをグローバルIPアドレスに変換する技術。PAT（Port Address Translation）は1つのグローバルIPをポート番号で区別して複数のプライベートIPで共有します。家庭用ルータのほぼ全てがPATを使用。

swap_calls NAT とは？

プライベートIP ↔ グローバルIPの「翻訳」機能（L3で動作）
  • 役割：RFC1918のプライベートIPをインターネット用グローバルIPに変換
  • 動作場所：ルータやファイアウォールで実施
  • 利点：IPv4アドレス枯渇問題を緩和、内部ネットワーク構造を隠蔽
  • 静的NAT：1対1でプライベートIPとグローバルIPを固定マッピング
会社の代表番号 — 外から見ると1つの番号、中では内線で振り分け

call_split PAT（NAPT）とは？

1つのグローバルIPを複数ホストで共有（ポート番号で区別）
  • 別名：NAPT（Network Address Port Translation）/ IP Masquerading
  • 仕組み：送信元IPを変換 ＋ ポート番号でセッションを一意に識別
  • 変換テーブル：プライベートIP:ポート ↔ グローバルIP:変換ポート をマッピング保存
  • 容量：ポート番号は16ビット(0〜65535)なので理論上最大約65535セッション共有可能
郵便局の私書箱 — 1つの住所に複数人の荷物を宛先番号(ポート)で区別

lock TLS/SSL — 暗号化通信の仕組み

TLS（Transport Layer Security）はHTTPS通信を保護する暗号化プロトコル。「証明書による認証 → 鍵交換 → 暗号化通信」の流れで、盗聴・改ざん・なりすましの3つを防ぎます。TLS 1.3が現在の標準です。

enhanced_encryption TLS が提供する3つの保証

機密性・完全性・認証 — HTTPSを「安全」にする3本柱
  • 機密性（盗聴防止）：共通鍵暗号(AES-GCM)でデータを暗号化 → 中間者は読めない
  • 完全性（改ざん検知）：AEAD認証タグ付きで暗号化 → 1ビットでも変わると即検出
  • 認証（なりすまし防止）：CA署名の証明書でサーバの正体を証明
  • 使用：HTTPS(443), SMTPS(465), IMAPS(993), DoT(853)
封筒に手紙を入れて錠をかけ、封印のハンコ(CA)付き — 届いても中身は読めず改ざんも検知

key 2種類の暗号化の使い分け

公開鍵暗号（鍵交換のみ）＋ 共通鍵暗号（実際のデータ転送）
  • 公開鍵暗号（RSA / ECDSA）：公開鍵で暗号化、秘密鍵で復号。処理が重いため鍵交換のみに使用
  • 共通鍵暗号（AES-256-GCM）：同じ鍵で暗号化/復号。高速。ハンドシェイク後のデータに使用
  • ECDHE：楕円曲線ディフィー・ヘルマン — 前方秘匿性を提供する鍵交換方式
  • 前方秘匿性（PFS）：秘密鍵が漏れても過去の通信は解読不可（セッションごとに使い捨て鍵）
公開鍵=誰でも使える南京錠、秘密鍵=自分だけの解除キー。鍵交換後は高速な金庫(AES)に切替

public IPv6 — 次世代インターネットプロトコル

IPv4のアドレス枯渇を解決するため設計された128ビットアドレス体系。約340澗（3.4×10³⁸）個のアドレスを持ち、NATなしでエンドツーエンド通信が可能。現在Googleへのアクセスの約40%はIPv6経由です。

format_list_numbered アドレス形式と省略ルール

128ビットを16ビット×8グループ、コロン区切り16進数で表記
  • フル表記：2001:0db8:85a3:0000:0000:8a2e:0370:7334
  • 省略①：各グループの先頭ゼロを省略 → 2001:db8:85a3:0:0:8a2e:370:7334
  • 省略②：連続ゼログループを::で1度だけ省略 → 2001:db8:85a3::8a2e:370:7334
  • ループバック：::1（IPv4の 127.0.0.1 に相当）
  • 未指定：::（IPv4の 0.0.0.0 に相当）

category アドレス種類（IPv6にブロードキャストなし）

ユニキャスト / マルチキャスト / エニーキャスト の3種類
  • グローバルユニキャスト：2000::/3 — インターネット上のグローバルアドレス
  • リンクローカル：fe80::/10 — 同一リンク内のみ有効（NIC起動時に自動設定）
  • ユニークローカル：fc00::/7 — IPv4プライベートアドレスに相当
  • マルチキャスト：ff00::/8 — IPv4ブロードキャストの代替として使用
  • エニーキャスト：同一アドレスを複数ノードに割当、最近接ノードへ転送

lightbulb 移行期の現状 — デュアルスタック共存

現在のインターネットはIPv4とIPv6が共存。デュアルスタック（1つの機器が両方のIPを持つ）が主流の移行方式。
IPv6対応のOSは起動時にリンクローカルアドレス（fe80::/10）を自動設定。グローバルアドレスはルータのRA（Router Advertisement）から自動取得（SLAAC）。
トンネリング技術（6in4, Teredo）でIPv4網上にIPv6パケットを包んで転送することも可能。

lan VLAN — 仮想ローカルエリアネットワーク

VLAN（Virtual LAN）は物理的な配線を変えずに、論理的にネットワークを分割する技術。同一スイッチ内でも異なるVLANの端末は直接通信できません。IEEE 802.1Q規格でフレームにVLAN IDタグを付加します。

lan VLANとは？

物理配線を変えずにネットワークを論理分割する仮想化技術
  • 規格：IEEE 802.1Q — フレームにVLAN IDタグ（12ビット = 4096種）を追加
  • VLAN ID：1〜4094（0と4095は予約済み）で最大4094個のVLAN
  • 目的：ブロードキャストドメインの分割・セキュリティ向上・管理簡素化
  • VLAN間通信：別VLANとの通信にはL3スイッチまたはルータが必須
同じオフィスビル内でも、社員証の色（VLAN ID）によって入れる部屋が違う

cable ポートの種類 — アクセス vs トランク

端末接続はアクセスポート、スイッチ間はトランクポートを使用
  • アクセスポート：1つのVLANのみ割り当て。PCやサーバに接続。タグなしフレーム
  • トランクポート：複数VLANを同時転送。スイッチ間・ルータ間の接続に使用
  • 802.1Qタグ：フレームに4バイトのVLANヘッダ（TPID + TCI）を挿入
  • ネイティブVLAN：トランクポートでタグなしフレームに割当てるVLAN（デフォルト=1）

lightbulb VLANが解決する問題

ブロードキャストストーム防止：VLAN分割でブロードキャストドメインを限定し、ARPやDHCPのパケットが全端末に届くのを防ぐ。
セキュリティ：人事部と開発部を別VLANにすることで、物理的に同じスイッチでも相互アクセスを制御可能。
柔軟な管理：部署異動や増設時に物理配線変更なしでネットワーク変更できる（スイッチの設定変更のみ）。
