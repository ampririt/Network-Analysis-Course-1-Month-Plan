netstat — see every connection

Lists every network socket on the machine: what's listening, and who's connected to whom.

Common options
-aShow all sockets — both listening and active connections.
-nNumeric: don't resolve host/port names — faster and clearer.
-t / -uTCP only / UDP only (Linux).
-lOnly listening sockets (Linux).
-pShow the owning program + PID (Linux; needs sudo).
-rShow the routing table instead of sockets.
The everyday combo: netstat -an
$ netstat -an # all sockets, numeric
Proto Local Address Foreign Address (state)
tcp4 192.168.1.42.22 203.0.113.7.51324 ESTABLISHED
tcp4 *.443 *.* LISTEN
Prototcp4 / tcp6 / udp — protocol & IP version.
Local AddressYour side as ip.port. * = any interface.
Foreign AddressThe other side. *.* = waiting for anyone to connect.
💡  macOS netstat has no -p — use lsof -i to see the process. Modern Linux: ss -tlnp.

Deep dive · netstat · 2 of 2

netstat — states & how to analyze

What the State column means
LISTENA service is waiting for connections on that port.
ESTABLISHEDAn active, open connection — data can flow.
TIME_WAITRecently closed; the OS holds it briefly — normal, common under load.
CLOSE_WAITThe other side closed but your app hasn't — a leak if it piles up.
SYN_SENTTrying to connect, no reply yet — firewall, or host is down.
How to analyze it
# is my web server listening on 443?
$ netstat -an | grep '.443 ' | grep LISTEN
 
# how many clients connected right now?
$ netstat -an | grep .443 | grep -c ESTABLISHED
37
 
# which remote IPs are connected?
$ netstat -an | grep ESTAB | awk '{print $5}'
🔎  Analyze: growing CLOSE_WAIT = an app isn't closing sockets (a bug); piles of SYN_SENT = you can't reach the server; lots of TIME_WAIT = just busy, usually fine.

curl — talk to any URL

Fetch pages and APIs — and measure exactly what happens on the way.

Common options
-sSilent — hide the progress meter.
-o file / -OSave the body to a file / to its remote name.
-LFollow redirects (3xx) to the final page.
-IFetch headers only (a HEAD request).
-H "K: V"Add a request header (e.g. an API token).
-X POST / -dSet the method / send body data.
-u / -kHTTP auth (user:pass) / skip the TLS certificate check.
--max-time NGive up if the whole request takes longer than N seconds (--connect-timeout limits just the connect phase).
-w write-out variables
%{http_code}The status code — 200, 404, 500…
%{time_total}Whole request time, in seconds.
%{time_namelookup}Time spent on DNS.
%{time_connect}Time to open the TCP connection.
%{time_starttransfer}Time to the first byte (server think-time).
discard the body · or save it
# discard the body → just the metrics
$ curl -s -o /dev/null \

# OR save the body to a file
$ curl -s -o health.json https://api/health
$ cat health.json
{"status":"ok","db":"up"}
# -O keeps the URL's own filename · -L follows redirects

curl — read & analyze the result

What the status code means
2xxSuccess — 200 OK, 201 Created, 204 No Content.
3xxRedirect — 301/302 → add -L to follow it.
4xxYour request — 401 auth, 403 forbidden, 404 missing, 429 rate-limited.
5xxServer's fault — 500 error, 502/503 down or overloaded.
000No HTTP at all — DNS fail, timeout, refused, or TLS error.
Analyze: where is the slowness?
$ curl -s -o /dev/null -w "dns:%{time_namelookup} \
  conn:%{time_connect} ttfb:%{time_starttransfer} \
  total:%{time_total}\n" https://api.example.com
dns:0.02 conn:0.05 ttfb:0.17 total:0.18
 
# inspect the response headers:
$ curl -sI https://api.example.com | head
🔎  Read the timings: big dns = DNS problem · big conn = network/TLS · big ttfb = slow server. -sI shows headers like Server, Content-Type, Location.

Pipes in practice · type it yourself

Follow along: build a pipeline

Goal: "how many services am I listening on?" Start with the raw firehose, then add one stage at a time and watch the output shrink to a single answer.

terminal▶ follow along
$ netstat -an # ① raw
… 312 lines — far too much to read …
 
$ netstat -an | grep LISTEN # ②
tcp4 *.22 LISTEN
tcp4 *.80 LISTEN
tcp4 *.443 LISTEN
 
$ … | grep LISTEN | awk '{print $4}' # ③
*.22 *.80 *.443
 
$ … | grep LISTEN | wc -l # ④
3
 
$ … | grep LISTEN > listening.txt # ⑤
netstat -an① The raw firehose — every socket on the box, too much to eyeball.
| grep LISTEN② The pipe feeds that text to grep, which keeps only lines with "LISTEN".
| awk '{print $4}'③ awk prints just the 4th column (the address.port), dropping the rest.
| wc -l④ wc -l counts the lines → one number: your answer.
> listening.txt⑤ Redirect the result into a file instead of the screen (>> appends).
🧪 Try it: run each line, check the result, then add the next |. Windows: Get-NetTCPConnection | ? State -eq Listen | Measure-Object.

Deep dive · tee

tee — see it AND save it

A pipe sends output to one place. tee copies it — to the screen and a file at the same time.

a commandmakes output | tee filemakes a copy → 🖥 screenyou see it + 📄 fileit's saved
In plain words
  • 1Normally a pipe | hands output to just one next command.
  • 2tee shows the output on screen and writes a copy to a file.
  • 3Add -a to append (add to the end) instead of overwriting.
Easy examples
$ ls | tee files.txt # see it + save it
file1 file2 file3
$ cat files.txt # it's in the file too
file1 file2 file3
$ date | tee -a log.txt # append a line
🔐  To write to a root-owned file, > fails with sudo — use echo "…" | sudo tee /etc/hosts instead.

awk — grab a column

awk reads text line by line and splits each line into fields (columns) you can print.

alice   25   NYC
$1 = alice    $2 = 25    $3 = NYC    $0 = the whole line
In plain words
  • 1awk cuts each line into columns at the spaces.
  • 2$1 is the 1st column, $2 the 2nd … $0 is the whole line.
  • 3{print $2} prints just that column for every line.
  • 4-F',' splits on a comma instead of spaces — for CSV files.
Easy examples
$ echo "alice 25 NYC" | awk '{print $1}'
alice
$ echo "alice 25 NYC" | awk '{print $3}'
NYC
$ echo "a,b,c" | awk -F',' '{print $2}'
b

awk — pick lines & do a little math

Put a pattern before the { } and awk acts only on matching lines — exactly how our scripts pull out what they need.

In plain words
/word/{print $2}Only on lines that contain “word”, print column 2.
$2+0Turns text like “11.8 ms” into the number 11.8.
END{ … }Runs once at the end — perfect for a total or a count.
NR>1“Skip line 1” — handy to ignore a CSV header row.
Straight from our scripts
# netreport: the router's IP
$ route -n get default | awk '/gateway/{print $2}'
192.168.1.1
 
# latency-log: just the ping time, as a number
$ ping -c1 8.8.8.8 | awk -F'time=' '/time=/{print $2+0}'
11.8
 
# count established connections
$ netstat -an | awk '/ESTAB/{n++} END{print n}'
42

Add the first line: the shebang

One special line at the very top tells the computer how to run your file.

check.sh
#!/usr/bin/env bash

# no shebang — the .ps1 name says PowerShell

Store the host name once, then reuse it — so you only change it in one place.

Let the user choose: an argument

Make the script work for any host the user types — with a sensible default.

check.sh
#!/usr/bin/env bash

In plain words
  • 1An argument is a value the user types after the script's name.
  • 2Bash: ${1:-google.com} = use the 1st argument, or google.com if none.
  • 3PowerShell: param($Target = "google.com") does the same — an arg with a default.

Make decisions with if, and repeat work with loops — the moment a one-liner becomes a script.

macOS · Bash
# loop over a list, test each
for n in 1 2 3; do

# loop over a range, test each
foreach ($n in 1..3) {
  $IP = "10.1.1.$n"
  if (Test-Connection $IP -Count 1 -Quiet) {
    "$IP UP"
  } else {
    "$IP down"
  }
}

Making a decision: if / else

Run different commands depending on whether something worked — on macOS and Windows.

decide.sh
# is the internet working?
if ping -c1 google.com; then
  echo "✅ we are online"
else
  echo "❌ no connection"
fi
decide.ps1
# is the internet working?
if (Test-Connection google.com -Quiet) {
  "✅ we are online"
} else {
  "❌ no connection"
}
In plain words
  • 1if runs a command/test. If it works, the first block runs.
  • 2else is the backup — it runs only when the if failed.
  • 3Bash closes the block with then … fi; PowerShell wraps it in { … }.
  • 4Compare values: Bash [ "$n" -gt 3 ]  ·  PowerShell $n -gt 3.

Scripting basics · repeating work

Repeat something: loops

Run the same lines many times — without copy-pasting them.

loops.sh
# FOR: once for each item in a list
for site in google.com github.com; do
  ping -c1 "$site"
done
 
# WHILE: keep going while true
n=1
while [ "$n" -le 3 ]; do
  echo "try $n"; n=$((n + 1))
done
loops.ps1
# FOR: once for each item in a list
foreach ($site in "google.com","github.com") {
  Test-Connection $site -Count 1
}
 
# WHILE: keep going while true
$n = 1
while ($n -le 3) {
  "try $n"; $n++
}
In plain words
  • 1A for loop repeats once for each item in a list (here, two sites).
  • 2A while loop repeats as long as a test stays true.
  • 3Bash wraps the repeated lines in do … done; PowerShell uses { … }.
  • 4Same idea on both systems — only the punctuation changes.

Reuse code: functions

Give a block of commands a name, then call it as many times as you like.

functions.sh
# define a function (a named block)
check() {
  ping -c1 "$1"
}
 
# call it by name, as often as you want
check google.com
check github.com
functions.ps1
# define a function (a named block)
function Check($h) {
  Test-Connection $h -Count 1
}
 
# call it by name, as often as you want
Check google.com
Check github.com
In plain words
  • 1A function is a named block of commands you can reuse.
  • 2Define it once (check() { } or function Check { }), then call it by name.
  • 3The value you pass arrives inside as $1 (Bash) or your $h parameter (PowerShell).
  • 4Write it once, call it everywhere — no copy-pasting the same lines.

A subnet ping-sweep

Everything so far — variables, a loop, a conditional, output — in one reusable tool that reports which hosts on a /24 are alive.

sweep.sh
#!/usr/bin/env bash
NET="${1:-10.1.1}" # default subnet
for i in $(seq 1 254); do
  if ping -c1 -W1 $NET.$i >/dev/null 2>&1; then
    echo "UP $NET.$i"
  fi
done
sweep.ps1
param($Net = "10.1.1")
1..254 | ForEach-Object {
  $ip = "$Net.$_"
  if (Test-Connection $ip -Count 1 -Quiet) {
    "UP $ip"
  }
}

Run it & save the result: ./sweep.sh 192.168.1 > alive.txt  ·  .\sweep.ps1 192.168.1 > alive.txt

Service health monitor

Hit a URL, check the HTTP status, log a line whenever it's down — an uptime watchdog for cron or Task Scheduler.

health-check.sh
#!/usr/bin/env bash
URL="${1:-https://google.com}"
code=$(curl -sL -o /dev/null -w "%{http_code}" --max-time 5 "$URL")
ts=$(date '+%F %T')
if [ "$code" -ge 200 ] && [ "$code" -lt 400 ]; then
  echo "$ts OK $URL ($code)"
else echo "$ts DOWN $URL ($code)" | tee -a outage.log; fi
health-check.ps1
param([string]$Url = "https://google.com")
$ts = Get-Date -Format "yyyy-MM-dd HH:mm:ss"
try {
  $r = Invoke-WebRequest $Url -TimeoutSec 5 -UseBasicParsing
  "$ts OK $Url ($($r.StatusCode))"
} catch {
  "$ts DOWN $Url" | Tee-Object outage.log -Append
}
terminal▶ sample run
$ ./health-check.sh https://google.com
2026-06-17 09:48:53 OK https://google.com (200) # 301→200
PS> .\health-check.ps1 https://down.test
2026-06-17 09:49:10 DOWN https://down.test (000)
🎯 Use it for uptime monitoring. -L follows redirects, so any 2xx–3xx = healthy; 4xx/5xx or 000 → outage.log.

TCP port scanner

Loop a list of ports and report which are open — a quick "what's listening?" check (nc on macOS, Test-NetConnection on Windows).

portscan.sh
#!/usr/bin/env bash
HOST="${1:-127.0.0.1}"
PORTS="${2:-22 80 443 3306}"
for p in $PORTS; do
  if nc -z -G1 "$HOST" "$p" 2>/dev/null
  then echo "OPEN $HOST:$p"
  else echo "closed $HOST:$p"; fi
done
portscan.ps1
param([string]$Target = "127.0.0.1",
      [int[]]$Ports = 22,80,443,3306)
foreach ($p in $Ports) {
  $ok = (Test-NetConnection $Target -Port $p `
      -WarningAction SilentlyContinue).TcpTestSucceeded
  if ($ok) { "OPEN ${Target}:$p" }
  else { "closed ${Target}:$p" }
}
terminal▶ sample run
$ ./portscan.sh scanme.local "22 80 443"
OPEN scanme.local:22
PS> .\portscan.ps1 scanme.local 22,80,443
closed scanme.local:443
🎯 Use it for a "what's exposed?" check after a firewall change. OPEN = the port accepted a connection; closed = refused or silent.

Latency logger → CSV

Ping a target on a timer and append the round-trip time to a CSV — catch intermittent drops you'd never see live, then graph the file.

latency-log.sh
#!/usr/bin/env bash
TARGET="${1:-8.8.8.8}"; OUT="latency.csv"
echo "timestamp,target,rtt_ms" >> "$OUT"
while true; do
  rtt=$(ping -c1 "$TARGET" | awk -F'time=' '/time=/{print $2+0}')
  echo "$(date +%FT%T),$TARGET,${rtt:-timeout}" >> "$OUT"
  sleep 5
done
latency-log.ps1
param([string]$Target = "8.8.8.8")
$out = "latency.csv"
"timestamp,target,rtt_ms" | Add-Content $out
while ($true) {
  $r = Test-Connection $Target -Count 1 -EA SilentlyContinue
  $rtt = if ($r) { $r.Latency } else { "timeout" }
  "$(Get-Date -Format s),$Target,$rtt" | Add-Content $out
  Start-Sleep 5
}

One-shot network report

A snapshot of this machine's networking — host, IP, gateway, DNS, public IP — in one command. Capture it before and after a change to see what moved.

netreport.sh
#!/usr/bin/env bash
echo "=== Host ==="; hostname
echo "=== IP ==="; ipconfig getifaddr en0
echo "=== Gateway ==="; route -n get default \
  | awk '/gateway/{print $2}'
echo "=== DNS ==="; scutil --dns \
  | awk '/nameserver/{print $3}' | sort -u
echo "=== Public ==="; curl -s ifconfig.me
netreport.ps1
"=== Host ===" ; hostname
"=== IP ===" ; (Get-NetIPAddress -AddressFamily IPv4).IPAddress
"=== Gateway ==="; (Get-NetRoute '0.0.0.0/0').NextHop
"=== DNS ===" ; (Get-DnsClientServerAddress `
   -AddressFamily IPv4).ServerAddresses
"=== Public ===" ; Invoke-RestMethod ifconfig.me/ip
terminal▶ sample run
$ ./netreport.sh (or PS> .\netreport.ps1)
=== Host === macbook.local
=== IP === 192.168.1.42
=== Gateway === 192.168.1.1
=== Public === 203.0.113.55
🎯 Use it for ticket snapshots and before/after diffs — local IP, gateway, DNS, and the public IP you exit from (NAT).

Run a command across a fleet

Read a list of hosts and run the same command on each — over SSH on macOS, WinRM (Invoke-Command) on Windows.

fleet.sh
#!/usr/bin/env bash
CMD="${*:-uptime}" # all args = command
while read -r host; do
  [ -z "$host" ] && continue
  echo "===== $host ====="
  ssh -o ConnectTimeout=5 "$host" "$CMD" \
    || echo " (unreachable)"
done < hosts.txt
fleet.ps1
param([string]$Cmd = "Get-Uptime")
foreach ($h in Get-Content hosts.txt) {
  if (-not $h) { continue }
  "===== $h ====="
  try { Invoke-Command -ComputerName $h `
    -ScriptBlock ([scriptblock]::Create($Cmd)) }
  catch { " (unreachable)" }
}
terminal▶ sample run
$ ./fleet.sh "df -h /"
===== web01 ===== /dev/sda1 40G 18G 48% /
PS> .\fleet.ps1 "Get-Volume C"
===== db01 ===== (unreachable)
🎯 Use it for one-pass audits across many hosts (disk, uptime, configs). A 5-second timeout flags a dead box as (unreachable) instead of stalling.

When should it run? The cron schedule

macOS & Linux use cron — just five numbers that say when to run a command. Learn to read them and scheduling is easy.

the five time fields
* * * * *  command-to-run
│ │ │ │ │
│ │ │ │ └─ day of week (0–6, Sun=0)
│ │ │ └─── month (1–12)
│ │ └───── day of month (1–31)
│ └─────── hour (0–23)
└───────── minute (0–59)
0 8 * * *every day at 08:00
*/5 * * * *every 5 minutes
0 * * * *every hour, on the hour
0 9 * * 1every Monday at 09:00
30 2 1 * *02:30 on the 1st of each month
🕐  Read left → right: minute, hour, day, month, weekday. A * means “every”, a number pins it to one value, and */5 means “every 5th”.

plan the schedule

The Windows way: Task Scheduler

Same job, same 8 AM — Windows uses Task Scheduler instead of cron. Set it up once from PowerShell.

PowerShell (run once)
# 1) WHEN it runs
$t = New-ScheduledTaskTrigger -Daily -At 8am
# 2) WHAT it runs
$a = New-ScheduledTaskAction -Execute pwsh `
    -Argument "C:\check.ps1"
# 3) save it
Register-ScheduledTask "Check" -Trigger $t -Action $a
In plain words
  • 1A task has two parts: a trigger (when) and an action (what to run).
  • 2-Daily -At 8am is the same idea as cron's 0 8 * * *.
  • 3Register-ScheduledTask saves it — Windows now runs it for you.
  • 4Manage it later in the Task Scheduler app: history, disable, delete.
🍎  macOS / Linux: the cron line from Steps 1–3 does exactly the same job.

Habits that save you

Scripting best practices

A script runs faster than you can think — which means it makes mistakes faster too. These habits keep automation safe in production.

  • ✓Fail loudly. Start bash scripts with set -euo pipefail; check exit codes ($? / $LASTEXITCODE).
  • ✓Quote your variables. "$HOST" avoids breakage when values contain spaces or are empty.
  • ✓Dry-run first. Echo what would happen before running destructive commands for real.
  • ✓Never hard-code secrets. Read passwords/keys from env vars or a vault, not the script body.
  • ✓Be idempotent. Running twice should be safe — check state before you change it.
  • ✓Comment & log. Future-you needs to know why; write timestamps to a log file.
