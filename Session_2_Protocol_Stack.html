Layer 2 · Data Link

The Ethernet frame — 14 bytes of header, then the world.

The simplest of the headers — but every byte on a typical LAN is wrapped in one. Hover any field.

Destination MAC48 bits
Hardware address of the next-hop device on this link. Switches forward by this field.
Source MAC48 bits
Hardware address of the sending NIC. Used by switches to learn the MAC table.
EtherType16 bits
Tells the receiver what the payload is. 0x0800 = IPv4, 0x0806 = ARP, 0x86DD = IPv6.
Payload46 – 1500 B
The IP packet (or ARP, or other) carried by this frame.
FCS (trailer)32 bits
CRC checksum appended after the payload. Detects corruption on the wire; corrupted frames are silently dropped.
Total frame: ~64 to 1518 bytes on the wire (without VLAN tag) — header + payload + FCS.

Header AnatomyUDP · L4 Datagram
Layer 4 · Transport

The UDP header — 8 bytes, and not one byte more.

Where TCP spends 20+ bytes earning reliability, UDP spends 8 and trusts the application to cope. Two ports, a length, a checksum — done.

0 8 16 24 — 31
Source Port16
Sender's port — often ephemeral, and on a pure one-way stream it may even be 0 (unused).
Destination Port16
Where to deliver: 53 DNS, 67/68 DHCP, 123 NTP, 443 QUIC.
Length16
Length of header + data in bytes. Minimum 8 (header only).
Checksum16
Optional in IPv4, mandatory in IPv6. Covers header, data, and an IP pseudo-header. No retransmit if it fails — the datagram is just dropped.
Data (payload)variable
The application's bytes — a DNS query, a video frame, a game state update. Handed up the instant it arrives.
Endpoints (ports) Length Integrity Payload

Read it as: "Here is a datagram for port 53. Here is how long it is. Take it or leave it — I won't ask again."

LectureTransport choice · TCP vs UDP
Two philosophies of delivery

Reliable conversation, or fast fire-and-forget?

Connection-oriented

TCP (Transmission Control Protocol)

Sets up a connection (3-way handshake), numbers every byte, acknowledges receipt, retransmits losses, and controls flow. You pay in latency and overhead; you get a clean, ordered byte stream.

HTTP/1-2HTTPSSSHSMTPFTP
Connectionless

UDP (User Datagram Protocol)

No setup, no acknowledgements, no ordering — just address it and send. If a datagram is lost, it's gone. Perfect when "late" is worse than "missing" and the app handles its own recovery.

ReferencePorts in detail · demultiplexing
One IP, many doors

A port is a 16-bit door on the host.

A host has one IP but 65,536 ports (0 – 65,535). The destination port tells the OS which listening program should receive each packet — this sorting is called demultiplexing.

packet → dst :443
packet → dst :22
packet → dst :53
▶
Host 203.0.113.7 · OS routes by port
:443 → Web server (HTTPS)
:22 → SSH daemon
:53 → DNS resolver
Source vs destination port

The client picks a throwaway ephemeral source port (49152+) and aims at the server's fixed well-known destination port. On the reply, the two simply swap places.

Listening vs connecting

A server listens on one fixed port; each client connects from its own ephemeral port. The 4-tuple src ip:port → dst ip:port keeps thousands of connections to :443 distinct.

Think of a building IP address → the street address (which building) Port → the apartment number (which program answers inside) Both source & destination port are 16-bit fields in every TCP / UDP header.

A socket is one endpoint of a connection.

A socket bundles an IP address + a port number (over TCP or UDP) into a single addressable point — the handle a program reads from and writes to. The IP finds the host; the port finds the program running on it.

How it works · server

SocketsWhat they're for · types · ports
every networked program speaks through a socket

What sockets are used for.

Open a web page, send mail, join a video call, query a database — each opens a socket. The OS hands the program a socket like a file descriptor, then demultiplexes every arriving packet to the right socket by its destination port + 4-tuple. That's how one machine runs many services at once.

Stream socket · TCP · SOCK_STREAM

Connection-oriented, reliable, ordered byte stream — a handshake first, then guaranteed delivery.

Used for: web (HTTP/HTTPS), email, SSH, file transfer, databases — anything that must arrive intact and in order.

Datagram socket · UDP · SOCK_DGRAM

Connectionless, no handshake, no guarantee — fire and forget, just fast.

Used for: DNS, DHCP, VoIP, live video, online games — anything that prefers speed over perfect delivery.

Network portion · 24 bits

Shared by every device on the LAN — 192.168.1.0. Routers forward by this part; it answers "which network?"

Host portion · 8 bits

Unique to each device — .10. With 8 host bits there are 2⁸ − 2 = 254 usable hosts (minus network & broadcast). It answers "which device?"

Why split it? Routers steer packets using only the network part — one routing entry per network, not per device — then the local switch uses the host part to reach the exact machine. It's a postal address: the ZIP code / city (network) carries the letter across the country; the street & house number (host) finds the door.

IP DesignRecap · before we subnet
caution · two addresses you can not assign

Every block reserves two addresses.

Before subnetting, lock in these three facts. In any IPv4 network, the first and last addresses are taken — so the devices you can actually use are everything in between.

Network address

Host bits = all 0

Names the network itself, not a device. Example 172.16.0.0 (Class B). It's what routing tables point at.

…0000 0000
Broadcast address

Host bits = all 1

Reaches every host on the link at once. Example 192.168.1.255 (Class C). The highest address in the block.

…1111 1111
Usable hosts

2ⁿ − 2

n = number of host bits. Subtract 2 — the network and broadcast addresses are already spoken for.

/24 → 2⁸ − 2 = 254
That's why a /24 hosts 254, not 256. Hold onto the network-and-broadcast rule — subnetting is just choosing where the host/network boundary falls, then re-deriving these same two edges for every new, smaller block.
IP DesignFrom classful to classless
A short history · why the rules changed

Why we dropped Class A / B / C.

The classful system (1981)
Class Fixed prefix Hosts per network
A /8 ~16,777,214
B /16 65,534
C /24 254

Your only choices — and the jumps between them are enormous.

Problem 1 · massive waste

Need 300 hosts? A Class C (254) is too small, so you take a Class B — ~65,000 addresses wasted.

Problem 2 · exhaustion & routing bloat

Handing out whole B's burned IPv4 fast, and millions of separate Class C routes swelled the internet's routing tables.

The fix — CIDR (1993): drop the classes and let the boundary fall on any bit. Need 300 hosts? A /23 gives 510; a /22 gives 1022 — right-sized. Neighbouring blocks also aggregate into one route, shrinking routing tables. ▸ next, how the notation works.
IP DesignSubnet masks & CIDR notation
Drawing the boundary

CIDR just counts the network bits.

CIDR — Classless Inter-Domain Routing. It replaced the rigid Class A / B / C system: the mask is written as /n, the count of leading 1-bits — that is, how many bits name the network.

The mask is a run of 1s

A subnet mask is 32 bits: all 1s for the network, then all 0s for the host. /24 simply means "the first 24 bits are network."

Why classful addressing died

Classes A/B/C locked you to /8, /16, /24 — wildly wasteful. CIDR lets the boundary fall on any bit, so subnets fit the actual host count.

CIDR Subnet mask Network / host split
/8 255.0.0.0 N.H.H.H
/16 255.255.0.0 N.N.H.H
/24 255.255.255.0 N.N.N.H

See one of these as a source IP in a capture from the public internet? It's spoofed or misconfigured — these never leave the LAN unaltered.

NAT bridges them out

Network Address Translation rewrites a private source IP to the router's single public IP on the way out — and reverses it on every reply.

Why they exist

IPv4 has only ~4.3 billion addresses. Private reuse + NAT let billions of devices share a small pool of public ones.

Two more reserved blocks

127.0.0.0/8 loopback (this machine) · 169.254.0.0/16 APIPA (link-local, when no DHCP).

150 PCs from Sales, IT and Accounting all share the single network 192.168.10.0/24.

  • A broadcast meant for Sales (ARP, DHCP Discover…) is flooded to IT and Accounting too.
  • Wasted bandwidth and CPU on every unrelated host.
  • No isolation — one noisy or compromised host touches everyone.
The question

How do we make a Sales broadcast reach Sales only?

What is a broadcast? A frame addressed to everyone on the network (ff:ff:ff:ff:ff:ff, or IP .255) that every host must receive and process — ARP "who has this IP?", DHCP Discover, and more. The more hosts share one network, the more of this traffic everybody has to handle.

MethodHow to subnet · step by step
A repeatable recipe

Subnetting in four moves.

Count the need. How many subnets, and how many hosts in the largest one? Size for the biggest.
Pick the prefix. Find the smallest mask whose usable hosts cover the need: usable = 2^(host bits) − 2.
Walk the blocks. Subnets begin at multiples of the block size 256 / 2ⁿ in the affected octet — e.g. /26 → .0, .64, .128, .192.
Name the edges. Network = all host bits 0, broadcast = all host bits 1, usable = everything between, gateway usually the first usable.
Quick formula · /(24 + n)

Subnet size = 256 / 2ⁿ addresses
Usable hosts = size − 2
Subnets start at multiples of the size

Worked · /26 (n = 2)

size = 256 / 4 = 64 · usable 62 · subnets at .0 / .64 / .128 / .192.

A company must split one Class B block into a separate network for every department.

Without subnetting

One flat network holds 2¹⁶ − 2 = 65,534 hosts — all 30 departments share one broadcast domain. It works, but gives no separation at all.

Step 1 — how many bits label ≥ 30 departments?
2² = 42³ = 82⁴ = 16 2⁵ = 32 ✓
2⁴ = 16 is too few; 2⁵ = 32 ≥ 30 covers every department (with 2 spare for growth).
→ 5 subnet bits

Then — how many bits for the hosts?

Each department needs up to 1023 PCs. Count up the powers of 2:

2⁷ = 1282⁸ = 2562⁹ = 512 2¹⁰ = 1024 ?
Why 2¹⁰ does NOT work

Two addresses in every block are reserved — the network (all host bits 0) and the broadcast (all host bits 1). So 10 host bits give only 2¹⁰ − 2 = 1022 usable. We need 1023 → one short!

So add one more bit

With 11 host bits: 2¹¹ − 2 = 2046 usable ≥ 1023 ✓ — comfortably fits, with headroom.

IP DesignBest practices · planning & documentation
Design once, grow without renumbering

A good address plan is boring on purpose.

01 · Plan by function

One subnet per role

Each department or device class gets its own subnet — keep the blocks contiguous so routes & firewall rules stay simple.

02 · Size for growth

Round up, leave gaps

Pick a prefix above today's host count and reserve the next block — renumbering a live network is painful.

03 · Document it

Write the plan down

Keep an IPAM sheet: every subnet, mask, VLAN, gateway and purpose. Undocumented space is how outages happen.

Network designTopology · how devices connect
The shape of the wiring

Four ways to wire a network — topology.

★ Star

Every device to one central switch. Today's standard.

Bus

One shared backbone. Cheap, fragile, legacy.

Ring

Each node to the next in a loop. Token-passing.

Mesh

Many cross-links. Resilient, costly — backbones.

Why it matters for design: a switch (or a VLAN on it) joins all its ports into one broadcast domain — exactly what a subnet maps to. Put each department on its own switch / VLAN / subnet and you contain the broadcast traffic we sized in the last slides.
