P. 09
LectureWhat is network analysis?
10 / 23
Definition
Network analysis is the practice of capturing, inspecting, and interpreting the traffic crossing a network — to understand, debug, secure, or improve what is happening on the wire. — working definition for this course

What you actually look at

A capture is a recording of frames seen on a network interface. Each frame carries headers from multiple protocols stacked together (e.g. Ethernet + IP + TCP + HTTP). An analyzer like Wireshark dissects those bytes and shows you fields you can read.

Two kinds of capture

Live capture — observe traffic in real time on your own NIC.
PCAP / PCAPNG file — a saved capture you can re-open, share, replay, and study offline. The PCAP file format is the lingua franca of the field.

What it is not

Not the same as monitoring CPU, RAM, or app logs. Network analysis works one layer deeper — at the level of bytes traversing a link. If logs say "request failed," the capture explains exactly which packet didn't arrive.

Who does it

Network engineers, sysadmins, SREs, security analysts, DFIR responders, threat hunters, protocol developers, QA engineers — and, increasingly, anyone curious about what their devices are actually saying.

P. 10
LectureVocabulary primer
11 / 23
First the words, then the picture

A few terms we'll use all month.

Protocol

A set of agreed rules

An agreement on the format and order of messages so that two strangers across a wire can understand each other. HTTP, TCP, IP, ARP, DHCP are all protocols.

Packet

The unit of network data

A formatted chunk of bytes. Strictly: a Layer-3 (IP) unit. Colloquially used for any unit of network traffic. We'll be precise when it matters.

Frame · Segment · Datagram

Name changes by layer

Same data, different name per layer: frame (L2 Ethernet), packet (L3 IP), segment (L4 TCP), datagram (L4 UDP).

Header

Metadata at the front

Every protocol prepends a small block of fields — where from, where to, how long, what kind — so receivers know how to interpret what follows.

Payload

The thing being carried

The data the higher layer handed down. One layer's payload is another layer's everything — Russian dolls all the way down.

MAC / IP address

Two ways to be reachable

MAC — burned-in 48-bit hardware ID for one NIC on one link.
IP — assigned, routable address across the wider network.

P. 11
ConceptWhat is a protocol? · 通信プロトコル
The single most important word in networking

A protocol is an agreement
between two strangers about how to talk.

Two devices that have never met before, possibly built by different vendors, possibly running different operating systems — must still exchange data and understand each other. A protocol is the written rulebook that makes this possible.

Pillar 01 · Syntax

The format of messages

Which bits mean what. Where the header ends and the payload begins. How long each field is. Without syntax, the receiver sees noise.

Pillar 02 · Semantics

The meaning of fields

What each field does — "this is an acknowledgement," "this is a destination," "this is a request to close." Same bits could mean different things in different protocols.

Pillar 03 · Timing

The order & timing of messages

Who speaks first. How long to wait. When to retry. When to give up. A protocol is a choreography, not just a vocabulary.

Every protocol you'll meet (HTTP, TCP, IP, Ethernet, ARP, DNS, …) is an agreed rulebook, published openly by the IETF as an RFC, so devices from completely different vendors can still talk.

P. 12
LectureWhy a layered model
13 / 23
A practical principle

Layers exist so we can
change one thing.

Separation of concerns

Each layer needs only to know how to talk to the layer above and below. The Wi-Fi card doesn't need to know it's carrying HTTP; HTTP doesn't need to know it's flying over fibre.

Independent evolution

Replace copper with fibre — the IP layer doesn't change. Invent QUIC — Ethernet doesn't change. The whole industry can innovate at one layer without breaking the others.

Diagnosis discipline

When something fails, asking "which layer broke?" is the single most useful question. Cable unplugged is L1. Wrong VLAN is L2. Bad route is L3. Firewall dropped it is L4. App returned 500 is L7.

A shared vocabulary

"It's a layer-2 problem" means the same thing to every engineer in every company. The OSI model is, more than anything, a way for humans to communicate about networks.

The model that actually shipped

OSI is the textbook. TCP/IP is what the internet runs.

The TCP/IP model (a.k.a. the DoD or Internet model) collapses the OSI 7 into 4 practical layers. It is what every real protocol implementation maps to.

OSI · 7 Layers

L7 · Application HTTP
L6 · Presentation TLS
L5 · Session RPC
L4 · Transport TCP/UDP
L3 · Network IPv4/v6
L2 · Data Link Ethernet
L1 · Physical Cat6/Fiber

TCP/IP · 4 Layers

Application HTTP · DNS · TLS · SMTP
Transport TCP · UDP · QUIC
Internet IP · ICMP · ARP*
Network Access (Link) Ethernet · 802.11 · MAC · PHY

* ARP is a famous edge case — operates between L2 and L3 and is usually grouped with the Link/Internet layer.

Reason 01 · Pragmatism

L5–L7 overlap in real code

Most apps just open a socket and write HTTP. Session & Presentation (state, encoding, TLS) live inside the app or its libraries — not separate kernel layers.

Reason 02 · Pragmatism

L1 + L2 ship together

A NIC driver handles both signalling bits and addressing frames. To the OS, "the network interface" is one thing — TCP/IP calls it one layer: Network Access.

Reason 03 · History

TCP/IP shipped first

OSI was the careful committee standard. TCP/IP came from working DARPA code people could already use. By the time OSI was ready, the internet was already TCP/IP.

Bottom line: use OSI to talk and reason; use TCP/IP to build and operate. (PPTX p.15 / textbook p.65)

Layer 2 · Data Link

The Ethernet frame — 14 bytes of header, then the world.

The simplest of the headers — but every byte on a typical LAN is wrapped in one. Hover any field.

Destination MAC48 bits
Hardware address of the next-hop device on this link. Switches forward by this field.
Source MAC48 bits
Hardware address of the sending NIC. Used by switches to learn the MAC table.
EtherType16 bits
Tells the receiver what the payload is. 0x0800 = IPv4, 0x0806 = ARP, 0x86DD = IPv6.
Payload46 – 1500 B
The IP packet (or ARP, or other) carried by this frame.
FCS (trailer)32 bits
CRC checksum appended after the payload. Detects corruption on the wire; corrupted frames are silently dropped.
Total frame: ~64 to 1518 bytes on the wire (without VLAN tag) — header + payload + FCS.
Addressing Demultiplex (what's inside?) Payload

Read it as: "I am sending from aa:bb:cc:11:22:33 to aa:bb:cc:44:55:66, and what comes next is an IPv4 packet."

The layer a device operates at determines what it can see, what it can decide, and what it must ignore. Recognising the layer is the first step in any troubleshooting question.

Concept Primer Before the binary · IP address as a postal address
Think of it like sending a letter

An IP address is the postal address of a device on the internet.

Postal addresses are hierarchical — country, then city, then street, then house number — so a letter can be routed in stages by progressively-narrower delivery offices. IP addresses work exactly the same way. The structure is what lets a router 12,000 km away know roughly where your packet needs to go without knowing exactly where you live.

Routers do exactly this with the IP address — each router only needs to know how to reach the next-larger network, not every individual device. Hierarchy

Country → city → street → house. In IP: network → subnet → host. The leftmost bits are the "country."

Reassignable

Move house → new postal address. Connect to a new Wi-Fi → DHCP gives you a new IP. The address belongs to where you are, not who you are.

Globally unique-ish

Two homes can't share one postal address. Two devices on the public internet can't share one public IP (NAT makes private IPs invisible from outside).

Switches learn MACs · "MAC table"

When a switch sees a frame from AA:BB:CC on port 3, it records "AA:BB:CC lives on port 3." Next frame headed to AA:BB:CC is forwarded only to port 3 instead of all of them. This is why LANs scale.

⚠ MACs can be spoofed in software — never trust a MAC as a security boundary. Use 802.1X / port security if it matters.

Concept Two addresses, two jobs · why we can't pick one
Same packet, two different addresses — and both are needed

Why does the internet use IP and MAC together?

They are not redundant — they answer two genuinely different questions. IP answers "where in the world is this device?" so routers across continents can route. MAC answers "which NIC on this exact wire should grab the frame?" so the local switch can deliver it. Lose either, and the internet stops working.

public
IP Address
Layer 3 · Network · logical · routable
"Where in the world is this device?"
  • Hierarchical — first bits identify the network, last bits the host. Routers can summarise.
  • Reassignable — your laptop gets a new IP every time you connect to a different network.
  • Routable across the planet — IP works across millions of intermediate links.
  • Not hardware-bound — software-assigned at the OS level.
memory
MAC Address
Layer 2 · Data Link · physical · local
"Which NIC on this wire should accept the frame?"
  • Flat (no hierarchy) — every address is its own thing. Globally unique by manufacturer assignment.
  • Burned in — assigned by the vendor at manufacture. Travels with the hardware.
  • Local-link only — never forwarded across routers. Re-written at every hop.
  • Not routable — a router cannot "find" a MAC across the internet.
"Why not just use MAC for everything?"
MAC addresses are flat — no structure. A router would need to know every one of the ~14 billion MACs on Earth individually. Routing tables wouldn't fit in memory, and updates would never converge. No hierarchy = no scale.
"Why not just use IP for everything?"
The wire only carries electrical / optical / radio signals — it has no concept of IP. The physical NIC needs a local hardware address to decide "is this frame for me?" without having to parse the IP header. No MAC = no local delivery.

The arrangement: IP gives the packet a stable end-to-end identity that survives every hop. MAC gives every hop a fast, hardware-level way to hand the packet to the next neighbour. The next slide shows this happening live — IP stays constant, MAC changes at every router.

Why between L2 and L3?

ARP carries L3 information (an IP address) but is encapsulated directly in an L2 frame — it skips IP entirely. That's why textbooks call it "Layer 2.5."

How a NIC recognises it

The Ethernet header's EtherType field is set to 0x0806 instead of 0x0800 (IPv4) — so the NIC hands it to the ARP code path, not the IP stack.

Where it works · where it doesn't

Only inside one broadcast domain (one LAN segment). Routers do not forward ARP. For IPv6, this job is done by NDP (Neighbor Discovery Protocol) instead.

Protocol SpotlightAddress Resolution Protocol · ARP
The IP-to-MAC bridge

ARP — "I know the IP, but I need the MAC."

Ethernet (L2) needs a MAC address to deliver a frame, but the application only knows the destination IP. ARP fills the gap by asking everyone on the local link.

Step 1 · ARP Request

Broadcast to everyone

PC A shouts on the local link:

"Who has 192.168.1.20?
Tell 192.168.1.10."

Destination MAC: FF:FF:FF:FF:FF:FF (broadcast — every NIC on the link receives it).

Step 2 · ARP Reply

The right host answers

PC B, recognising its own IP, replies directly:

"That's me — 192.168.1.20.
My MAC is aa:bb:cc:44:55:66."

Sent as unicast straight back to PC A. The pair is cached so future packets skip the broadcast.

Related: RARP (Reverse ARP) does the opposite — given a MAC, what's my IP? Largely replaced today by DHCP.
