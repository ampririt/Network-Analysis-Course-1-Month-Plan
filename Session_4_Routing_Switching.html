ConceptLayer 2 vs. Layer 3
Where switching ends and routing begins

Switches use MACs. Routers use IPs.

A frame moving inside one network is a Layer-2 problem, solved with MAC addresses. A packet crossing between networks is a Layer-3 problem, solved with IP addresses. The two jobs define two devices.

Layer 2 · Switch

Same network, frame by frame

Forwards Ethernet frames by destination MAC. One switch = one broadcast domain. No idea what an IP subnet is.

Layer 3 · Router

Network to network, hop by hop

Forwards IP packets by destination subnet. Each interface is a different network; a router is a broadcast-domain boundary.

Two domains to keep straight: a collision domain is a single shared-media segment (every switch port is its own); a broadcast domain is everywhere a broadcast reaches — bounded only by a router (or a VLAN).
CompareHub · Switch · Router
Three boxes, three jobs

Not all boxes are equal.

DeviceLayerForwards byCollision domainsBroadcast domains
HubL1 · physicalNothing — repeats bits to all ports1 (shared)1

A hub is "dumb" — everyone hears everything and collides. A switch gives every port its own collision domain. A router is the only one of the three that breaks a broadcast domain.
— Session 04 / Part 1 —

How a switch
moves frames.

One table, three decisions: learn the source, forward to the known port, or flood when you don't know yet.

SwitchingThe MAC address table
The switch's one source of truth

Every frame triggers one decision.

A switch keeps a MAC address table (CAM table) mapping each known MAC to the port it lives behind. For every incoming frame it does two things — learn, then forward.

1 · Learn

Source MAC → port

Record the frame's source MAC against the port it arrived on. This is how the table fills.

2 · Forward (known)

Unicast to one port

If the destination MAC is in the table, send the frame out only that port.

3 · Flood (unknown)

Out every other port

If the destination is unknown (or a broadcast), flood it out all ports except the one it came in on.

Aging: entries expire after an idle timeout (default 300 s on Cisco) so the table tracks devices that move. show mac address-table shows it live.

SwitchingUnicast · Broadcast · Multicast
One frame, three audiences

Who is this frame for?

Unicast

Exactly one

A specific destination MAC. Once learned, the switch sends it to a single port.

Broadcast

Everyone on the LAN

Destination ff:ff:ff:ff:ff:ff. Flooded to the whole broadcast domain — ARP, DHCP Discover.

Multicast

A subscribed group

One-to-many for those who opted in — video, routing protocols (OSPF 224.0.0.5).

Unknown-unicast flooding: a unicast frame whose destination isn't in the table yet is treated like a broadcast — flooded everywhere until the switch learns the reply. Too many of these is a red flag in a capture.
— Session 04 / Part 2 —

One switch,
many networks.

VLANs slice a single physical switch into separate broadcast domains — for security, for order, and to stop broadcasts from flooding the whole building.

VLANsWhy segment a network?
A VLAN = a broadcast domain you define in software

One wire, separate worlds.

Without VLANs, every port on a switch shares one broadcast domain. A VLAN lets you group ports logically — Admin, Sales, IT — even though they share the same hardware.

Security

Isolation by design

Devices in VLAN 10 can't reach VLAN 20 at Layer 2 at all — traffic must go through a router that can filter it.

Performance

Smaller broadcast domains

A broadcast in Sales never floods Admin. Less noise, less wasted bandwidth on every host.

Organisation

Logical, not physical

Group by role or department regardless of which floor or switch a user is plugged into.

In the lab: VLAN 10 = Admin, VLAN 20 = Sales, VLAN 30 = IT — three isolated networks on two shared switches.
VLANsWhy segment — in depth
One flat network becomes a liability as it grows

Six reasons to segment.

shieldSecurity & isolation

L2 traffic can't cross VLANs at all. Finance, servers and guests are walled off — inter-VLAN traffic must pass a router/firewall where it can be filtered.

cell_towerBroadcast containment

Each VLAN is its own broadcast domain. An ARP or DHCP storm in Sales never touches Admin, and every host wastes less CPU on broadcasts.

speedPerformance

Smaller domains mean less flooding and less unknown-unicast noise — more of the wire is left for real traffic.

groupsGrouping & mobility

Group by role, not location. A user keeps their VLAN when they move desks or floors — no recabling, just a port config.

savingsCost & simplicity

One physical switch does the work of many separate ones — no extra hardware per department.

tunePolicy & QoS

Apply ACLs, QoS and monitoring per VLAN — e.g. prioritise a Voice VLAN, throttle a Guest one.

A VLAN is a broadcast domain you draw in software — but remember: a router (or L3 switch) is still required for two VLANs to talk.

Look for TPID 0x8100 in the Ethernet II header; the 12-bit VID is the VLAN ID. Filter: vlan.

The native VLAN

One VLAN per trunk travels untagged (default VLAN 1). Both ends must agree — a mismatch is a classic misconfig.

VLANsAccess vs. trunk ports
Two kinds of switch port

One VLAN, or all of them.

Access port

Faces an end device

Belongs to exactly one VLAN. The PC plugged in knows nothing about VLANs — frames arrive and leave untagged. This is where laptops, printers and servers connect.

switchport mode access
switchport access vlan 10

Trunk port

Carries many VLANs

Links switch-to-switch or switch-to-router. Frames cross 802.1Q-tagged so multiple VLANs share one cable. This is how VLAN 10 on switch 1 reaches VLAN 10 on switch 2.

switchport mode trunk
switchport trunk allowed vlan 10,20,30

Rule of thumb: access toward hosts, trunk between switches. Verify with show interfaces trunk.
VLANsSpanning Tree · why we need it
Ethernet has no TTL — a looped frame never dies

Why does a LAN need STP?

We want redundant links for reliability — but two paths between switches form a Layer-2 loop. An IP packet has a TTL that expires; an Ethernet frame does not. So a single broadcast circulates forever, multiplying at every switch.

lanSW-A
lanSW-B
cycloneLOOP
cycloneBroadcast storm

Broadcasts & unknown-unicast are flooded, loop back, and are flooded again — link use hits 100% and the LAN freezes in seconds.

swap_horizMAC table instability

The same source MAC keeps arriving on two ports — the CAM table "flaps", so forwarding decisions go wrong.

content_copyDuplicate frames

Hosts receive multiple copies of every frame, confusing upper-layer protocols.

That's why STP (802.1D) runs by default on every switch: it keeps the physical redundancy but logically removes the loop by blocking one port.

VLANsSpanning Tree · how the roles are calculated
Four decisions, always in this order

Lowest ID, then lowest cost.

badgeBridge ID — who can be root

Bridge ID = Priority + MAC
Priority defaults to 32768 (set in steps of 4096); the lowest Bridge ID wins. Tie on priority → lowest MAC.

speedPort cost — by link speed
10 Mbps1001 Gbps4
100 Mbps1910 Gbps2
1 · Root Bridge

Lowest Bridge ID in the whole network. Every other switch builds its tree toward it.

2 · Root Port (per switch)

Lowest Root Path Cost — the sum of port costs back to the root. Tie-break: lowest sender Bridge ID, then lowest port ID.

3 · Designated Port (per segment)

On each link, the port advertising the lowest Root Path Cost forwards. The root's ports are always Designated.

4 · Blocking

Every remaining port is put into blocking — no loop, ready as backup.

Worked example (the diagram): all links are 100 Mbps (cost 19). SW2 & SW3 each reach root at cost 19 → Root Ports. On SW2–SW3 the cost is tied, so SW2's lower Bridge ID makes it Designated and SW3 blocks. RSTP (802.1w) is the faster modern version.
— Session 04 / Part 3 —

Between
the networks.

A router's whole job is one question, asked for every packet: "which way is the destination network, and what's the next hop?"

RoutingThe routing table
A map of reachable networks

Match the longest prefix.

For each packet the router scans its table for entries whose network matches the destination IP, then picks the most specific one — the longest prefix. That entry's next-hop / exit interface is where the packet goes.

DestinationNext hop / interfaceSourcePrefix
192.168.10.0/24G0/0.10 (connected)C — direct/24
192.168.20.0/24G0/0.20 (connected)C — direct/24
10.0.0.0/8via 203.0.113.1S — static/8
0.0.0.0/0via 203.0.113.1S* — default/0
A packet to 10.5.5.5 matches both 10.0.0.0/8 and 0.0.0.0/0 — the /8 wins because it's more specific. View it with show ip route.
RoutingStatic · dynamic · default
Where routes come from

Type it, or learn it.

Static

You configure it

Manual, predictable, zero protocol overhead — but you maintain every route by hand. Great for small or stub networks.

Dynamic

Routers learn it

Protocols (RIP, OSPF) share reachability and adapt to failures automatically — essential as the network grows.

Default route

"Everywhere else"

0.0.0.0/0 — the gateway of last resort. If nothing else matches, send it here (usually toward the internet).

When two sources offer the same route, who wins? Administrative distance ranks the source (connected 0 < static 1 < OSPF 110 < RIP 120); the metric then picks the best path within a protocol.
RoutingDynamic protocols
Two ways a router learns the map

Distance-vector vs. link-state.

RIP · distance-vector

"Routing by rumour"

Tells neighbours its whole table periodically. Metric is just hop count (max 15). Simple, but slow to converge and blind to link speed.

OSPF · link-state

Everyone builds the map

Every router floods link-state info, then each computes the shortest path (Dijkstra) over cost (based on bandwidth). Fast convergence, scales to large networks.

The headline difference: RIP knows only "how far", OSPF knows "the whole topology". That's why OSPF reacts to a failed link in seconds while RIP can take minutes. (Homework dives into OSPF.)
RoutingInter-VLAN routing
VLANs isolate — so how do they ever talk?

They can't, until a router joins them.

VLANs are separate networks, so VLAN 10 reaching VLAN 20 is a routing problem. Two ways to solve it:

Router-on-a-Stick

One trunk, many sub-interfaces

A single router link carries every VLAN. The physical port is split into sub-interfaces — one gateway IP per VLAN — each tagged with encapsulation dot1Q.

interface g0/0.10
 encapsulation dot1Q 10
 ip address 192.168.10.1 255.255.255.0

Layer-3 switch (SVI)

Routing inside the switch

A multilayer switch routes between VLANs itself using Switched Virtual Interfaces (interface vlan 10). No external router hop — faster, the standard in real campuses.

ip routing
interface vlan 10
 ip address 192.168.10.1 255.255.255.0

RefresherSubnetting
The address math behind every gateway

Network part, host part.

The subnet mask (or /CIDR) splits an IP: the network portion identifies the subnet, the host portion identifies the device. The router's job is purely about that network portion.

Each lab VLAN is its own /24: Admin 192.168.10.0, Sales .20.0, IT .30.0 — 254 hosts each, gateway on .1.
GlueFirst-hop & address services
Tying L2 and L3 together

Getting to the first hop.

ARP at the boundary

IP → MAC, every hop

To leave its subnet a host ARPs for its default gateway's MAC, not the far destination. Each router then ARPs for the next hop. (Recall Session 3.)

DHCP relay

One server, many subnets

DHCP Discover is a broadcast — it can't cross a router. An ip helper-address on the gateway relays it as unicast to a central DHCP server.

In the Mega-Lab the router itself is the DHCP server for all three VLANs — so each sub-interface hands its VLAN the right gateway, mask and DNS.

GotchasWhere the lab usually breaks
Learn these and the ping just works

The usual suspects.

Forgotten encapsulation

Sub-interface with no dot1Q tag (or the wrong VLAN ID) — the gateway is unreachable.

Trunk not allowing VLAN

A VLAN missing from trunk allowed never crosses between switches.

Native VLAN mismatch

Different native VLAN on each trunk end — frames leak between VLANs or get dropped.

Wrong / missing gateway

PC has no default gateway (or not the .1) — it can ping its own VLAN but nothing beyond it.

Access port wrong VLAN

Host assigned to the wrong VLAN gets a DHCP lease from the wrong pool — or none.

No default route

Inter-VLAN works but the internet doesn't — there's no 0.0.0.0/0.
