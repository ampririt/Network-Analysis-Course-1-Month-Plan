**学習目標**: 機器がどうIPを得て、名前を解決し、MACアドレスを発見するかを理解する。これらのプロトコルをキャプチャで解析する。

### 講義

**ネットワークサービス**
- *ネットワークサービス*とは、ある機器（**サーバ**）が他（**クライアント**）にネットワーク越しに提供する機能で、**IP + トランスポートプロトコル + ポート**でアドレス指定されます（例 DNS = `udp/53`、DHCP = `udp/67/68`）。
- これらはセッション1〜2のプロトコルスタックの上に乗り、「ネットワークが動く」基盤です: LANに参加する機器はアドレスを得（**DHCP**）、次ホップのMACを見つけ（**ARP**）、名前を解決し（**DNS**）、インターネットに到達します（**NAT**）。
- **ARP**: Address Resolution Protocol — MAC発見、ARPテーブル、gratuitous ARP
- **DHCP**: Discover → Offer → Request → Acknowledge（DORA プロセス）
- **DNS**: クエリと応答、レコードタイプ（A, AAAA, CNAME, MX, TXT）、再帰 vs 反復
- **ICMP**: Echo 要求/応答（`ping`）、TTL超過・到達不能メッセージ、traceroute の仕組み
- **NAT**: Network Address Translation — プライベートIPがどうインターネットに届くか
- **ダブルNAT**: 積み重なったNAT（例 自分のルータの背後にISPルータ）— 症状とポートフォワーディングが壊れる理由
- **HTTP/HTTPS と TLS**: アプリケーション層のWebトラフィックと暗号化 — *巨大な話題で、アプリケーション層専用のセッションに値する*
- **SSH / Telnet**: リモート管理 — *セキュリティのセッションと相性が良い*
- **SMTP / IMAP / POP3**: メールプロトコル — *メールフローに関心がある場合のみ*

**ネットワークセキュリティ**
- **なぜサービスが標的か**: 主要サービス（ARP, DHCP, DNS）の多くは*組み込みの認証がない*ため、なりすましや悪用が容易。
- **CIA トライアド**: 機密性（Confidentiality）・完全性（Integrity）・可用性（Availability）— あらゆるセキュリティ制御が守る目標。
- **サービスへの一般的攻撃**:
  - **ARP スプーフィング／ポイズニング** → 中間者攻撃（MITM）、トラフィック傍受
  - **不正DHCPサーバ** → 悪いゲートウェイ/DNSを配って被害者を誘導
  - **DNS スプーフィング／キャッシュポイズニング** → 悪意あるIPへ誘導
  - **MAC フラッディング／ポートスティーリング** → スイッチのCAMテーブルを溢れさせる
  - **DoS / DDoS** → サービスを溢れさせてリソースを枯渇
- **偵察とスキャン**: ポートスキャン、pingスイープ、OSフィンガープリンティング（例 Nmap）— 攻撃の*前*に攻撃者が行うことと、それがキャプチャでどう見えるか。
- **防御**: スイッチの堅牢化（**DHCP スヌーピング**、**Dynamic ARP Inspection**、**ポートセキュリティ**）、暗号化（**HTTPS/TLS**、Telnet より **SSH**）、ファイアウォールとセグメンテーション（**VLAN**、**DMZ**）、**DNSSEC**。
- **アナリストの心構え**: 正常トラフィックの*ベースライン*を確立し、異常（予期しないARP応答、スパイク、奇妙なポート）が目立つようにする。

### 詳解：主要ネットワークサービスリファレンス

#### ウェルノウンポートとトランスポート

| サービス | トランスポート · ポート | 役割 |
|:---|:---|:---|
| **DHCP** | UDP **67**（サーバ）/ **68**（クライアント） | IP・マスク・ゲートウェイ・DNSを配る |
| **DNS** | UDP/TCP **53** | 名前 → IP の解決 |
| **HTTP** | TCP **80** | Web（平文） |
| **HTTPS** | TCP **443** | Web（TLS暗号化） |
| **FTP** | TCP **21**（制御）/ **20**（データ） | ファイル転送（平文） |
| **SSH / SFTP** | TCP **22** | セキュアなリモートシェル＋ファイル転送 |
| **Telnet** | TCP **23** | リモートシェル（平文 — 避ける） |
| **SMTP** | TCP **25** / **587** | メール送信 |
| **POP3 / IMAP** | TCP **110** / **143** | メール受信 |
| **NTP** | UDP **123** | 時刻同期 |
| **SNMP** | UDP **161 / 162** | 機器監視 |
| **ARP** | *なし*（第2層） | ローカルリンクでの IP → MAC |
| **ICMP** | *なし*（第3層） | `ping`、エラー、traceroute |

> [!NOTE]
> **ARP と ICMP はポート番号を持ちません** — UDP/TCP の中に運ばれないからです。ARP は Ethernet フレームに直接（第2層）、ICMP は IP パケットに直接（第3層）乗ります。

#### DNS レコードタイプ

| レコード | 対応づけ | 例 |
|:---|:---|:---|
| **A** | 名前 → **IPv4** | `gaia.cs.umass.edu → 128.119.245.12` |
| **AAAA** | 名前 → **IPv6** | `… → 2607:f600:1002:6113::100` |
| **CNAME** | 名前 → 別の**名前**（別名） | `www → web.example.com` |
| **MX** | ドメイン → **メールサーバ** | `example.com → mail.example.com` |
| **NS** | ドメイン → その**権威ネームサーバ** | `umass.edu → ns1.umass.edu` |
| **PTR** | IP → 名前（**逆引き**） | `128.119.245.12 → gaia…` |
| **TXT** | 名前 → 自由**テキスト**（SPF、ドメイン認証） | `v=spf1 …` |
| **SOA** | ゾーンの **Start Of Authority** レコード | ゾーンの所有/提供者 |

#### DHCP DORA と主要オプション

| ステップ | メッセージ | 方向 | 運ぶもの |
|:---:|:---|:---|:---|
| 1 | **D**iscover | クライアント → ブロードキャスト | xid、**Parameter Request List** |
| 2 | **O**ffer | サーバ → クライアント | 提示IP（`yiaddr`）+ オプション |
| 3 | **R**equest | クライアント → ブロードキャスト | **Option 50**（要求IP）、**Option 54**（サーバID） |
| 4 | **A**CK | サーバ → クライアント | リース + 下記オプション |

**ACK** で配られる一般的オプション:

| オプション | 名称 | クライアントが受け取るもの |
|:---:|:---|:---|
| **1** | Subnet Mask | サブネットサイズ |
| **3** | Router | デフォルトゲートウェイ |
| **6** | Domain Name Server | DNSリゾルバ |
| **51** | IP Address Lease Time | アドレスの有効期間 |
| **53** | DHCP Message Type | Discover / Offer / Request / ACK |

> [!TIP]
> 4つのDORAメッセージは1つの **Transaction ID（xid）** を共有します — 複数のやり取りが同時進行でも、クライアントが Offer/ACK を自分の Discover/Request と対応づけられる仕組みです。

### 詳解：攻撃と防御

主要サービス（ARP, DHCP, DNS）は**組み込みの認証なし**で出荷されるため、どれもなりすましが容易です。各攻撃を、それを止める制御と、ワイヤ上での見え方に対応づけます:

| 攻撃 | 何をするか | キャプチャでの兆候 | 防御 |
|:---|:---|:---|:---|
| **ARP スプーフィング／ポイズニング** | ゲートウェイのIPを名乗る → MITM | **1つのIPに2つのMAC**、gratuitous ARP の洪水 | **Dynamic ARP Inspection（DAI）** |
| **不正DHCPサーバ** | 悪いゲートウェイ/DNSを配る | **予期しないサーバからのOffer** | **DHCP スヌーピング**（信頼ポートのみ） |
| **DNS スプーフィング／キャッシュポイズニング** | 応答を偽造 → 誤ったIP | **不一致/偽造アドレス**の応答 | **DNSSEC**、検証するリゾルバ |
| **MAC フラッディング** | スイッチのCAMテーブルを溢れさせる → フラッド | **新規送信元MACの嵐** | **ポートセキュリティ**（ポートあたり最大MAC数） |
| **平文の盗聴** | 通信中の資格情報を読む | Follow-Stream で**読める パスワード** | **SSH / SFTP / HTTPS**（暗号化） |
| **DoS / DDoS** | サービスを溢れさせ枯渇させる | 1つ/多数からの大量トラフィック | レート制限、ファイアウォール、スクラビング |

> [!NOTE]
> **アナリストの心構え:** まず*正常*の姿 — **ベースライン** — を学ぶこと。異常（予期しないARP応答、2台目のDHCPサーバ、連番ポートへのSYNのスパイク）は、そのベースラインと*比べて*こそ目立ちます。
